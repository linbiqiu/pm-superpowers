# 中文输出规范

PM Superpowers 面向中国团队使用。除非用户明确要求英文或双语，所有对用户可见的输出都必须以中文为核心语言。

## 默认语言

- 用中文输出结论、解释、表格标题、字段名、模板标题和交付产物。
- 插件名、技能名、文件名、代码标识、数据字段、专有产品名、英文框架名可以保留英文。
- 英文框架第一次出现时，优先补中文解释，例如 `RICE（覆盖人数、影响、信心、投入）`。
- 不要把内部英文术语原样作为主要标题，例如不要用 `Gate Review`，应使用 `门禁检查`。

## 写作风格

- 面向产品经理和业务同事，使用通俗、直接、可执行的中文。
- 避免过度学院化表达。
- 避免只给抽象框架，要结合用户当前场景解释“为什么”和“下一步怎么做”。
- 当信息不足时，用中文明确写出 `BLOCKED`，并说明阻断原因和需要补齐的信息。
- 当可以继续但有风险时，用中文明确写出 `PASS_WITH_RISKS`，并列出风险。

## 标准输出结构

优先使用以下中文结构：

```markdown
## 当前判断

- 场景：
- 输入质量：
- 门禁状态：
- 推荐下一步：

## 已知信息

## 关键假设

## 证据与信心

## 阻断项或风险

## 建议工作流

## 标准产物

## 下游交接提醒
```

## 术语约定

| 英文术语 | 中文表达 |
| --- | --- |
| Gate Review | 门禁检查 |
| Evidence | 证据 |
| Assumption | 假设 |
| Product Brief | 产品简报 |
| PRD | 产品需求文档 |
| Roadmap | 路线图 |
| Outcome Roadmap | 结果导向路线图 |
| Acceptance Criteria | 验收标准 |
| Downstream Readiness | 下游交接就绪检查 |
| Launch Readiness | 上线准备检查 |
| Post-launch Learning | 发布后学习 |
| Stakeholder | 相关方 |
| Owner | 负责人 |
| Open Questions | 待确认问题 |

## 禁止事项

- 不要默认输出全英文模板。
- 不要用英文标题包住中文内容。
- 不要为了显得专业而堆术语。
- 不要把未验证假设写成确定事实。
- 不要在信息不足时强行产出完整 PRD、路线图或上线计划。
