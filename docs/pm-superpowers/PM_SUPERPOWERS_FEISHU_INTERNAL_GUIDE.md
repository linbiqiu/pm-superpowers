# PM Superpowers 产品经理工作流助手：安装与场景使用指南

这份文档给内部产品经理使用。每个要使用 PM Superpowers 的同事，都需要先完成一次安装；团队通知升级时，再按步骤更新。

安装和更新不需要理解技术原理，复制命令执行即可。装好以后，你只需要描述真实业务背景，PM Superpowers 会判断产品场景，并带你走对应工作流。

<callout emoji="📌" background-color="light-blue" border-color="blue">
  <p><b>先说结论：</b>先安装，再使用；先判断场景，再写产物；信息不清楚时，先补信息，不要硬往下走。</p>
</callout>

## 先安装并确认可用

先说明一点：PM Superpowers 已经放在 GitHub 上，但 Codex 不会自动搜索整个 GitHub。大家不用自己去搜索插件，直接复制下面的安装命令即可。

插件 Git 地址：`https://github.com/linbiqiu/pm-superpowers`

### 第一步：确认 Codex 可用

打开终端，执行：

```bash
codex plugin --help
```

能看到帮助信息就继续。
如果提示找不到 `codex`，联系团队工具负责人。

### 第二步：安装 PM Superpowers

复制执行下面两行。第一行是告诉 Codex 去 GitHub 上读取我们的 PM Superpowers 仓库，第二行是安装插件。

```bash
codex plugin marketplace add linbiqiu/pm-superpowers --ref main
codex plugin add pm-superpowers@pm-superpowers-internal
```

### 第三步：检查是否安装成功

执行：

```bash
codex plugin list
```

看到类似信息即成功：

```text
pm-superpowers@pm-superpowers-internal  installed, enabled
```

### 第四步：新开对话

安装后，新开一个 Codex 对话窗口。旧窗口通常不会自动加载新能力。

## 如何更新

团队通知有新版本时，复制执行：

```bash
codex plugin marketplace upgrade pm-superpowers-internal
codex plugin add pm-superpowers@pm-superpowers-internal
```

更新后，新开一个 Codex 对话窗口。

如果你本地已经下载过插件仓库，也可以在仓库根目录执行：

```bash
scripts/check_pm_superpowers_update.sh
scripts/update_pm_superpowers.sh
```

## 安装更新遇到问题怎么办

| 问题 | 处理方式 |
| --- | --- |
| 不知道终端在哪里 | 找团队工具负责人协助 |
| `codex` 命令找不到 | 说明 Codex CLI 没准备好 |
| GitHub 访问失败 | 换网络，或请工具负责人用 SSH/内部源安装 |
| 安装后看不到工具 | 新开 Codex 对话窗口 |
| 更新后还是旧行为 | 新开 Codex 对话窗口 |

## 它帮你做什么

PM Superpowers 是产品经理工作流助手，不是单纯的 PRD 生成器。

它会帮你：

- 判断当前属于哪个产品场景。
- 追问缺失信息。
- 输出中文产品产物。
- 检查是否能交给 UI、研发、测试或运营。
- 信息不足时提示 `BLOCKED`，避免把模糊需求传给下游。

## 第一次怎么开始

不要一上来就说“帮我写 PRD”。先让它判断场景。

可以这样说：

```text
我现在有一个产品需求，但还不确定应该怎么推进。请先帮我判断属于哪个产品场景，并告诉我继续前缺哪些信息。
```

如果已经有业务背景，就说清楚：

```text
我们想做一个给销售主管看的 AI 周报功能，目标是减少人工整理时间。请先帮我判断应该走哪个产品工作流，不要直接写 PRD。
```

## 开始前先回答五个问题

| 问题 | 为什么重要 |
| --- | --- |
| 这件事从哪里来？ | 判断是反馈、指标、老板想法、上线任务，还是战略方向 |
| 服务谁？ | 避免做出没人真正需要的需求 |
| 现在要做什么决策？ | 决定做不做、先做哪个、怎么做、能不能上线 |
| 最大不确定是什么？ | 找到需要先补的信息 |
| 产物给谁用？ | 决定输出给自己、设计、研发、销售客服还是管理层 |

<callout emoji="📌" background-color="light-yellow" border-color="yellow">
  <p><b>判断规则：</b>这五个问题里有两个以上答不清，就先做澄清，不要直接写 PRD。</p>
</callout>

## S0：模糊需求澄清

适合：只有一句想法、老板要求、竞品压力、客户随口提的诉求，还不知道下一步怎么走。

直接这样说：

```text
老板提到我们也要做 AI 助手，但目标用户和具体场景还不清楚。请先帮我做需求澄清，判断这属于哪个产品场景，并列出必须补齐的信息。
```

会得到：澄清后的任务、已知信息、关键假设、阻断项、推荐下一步。

先别继续：用户、问题、决策目标都不清楚时，不要写 PRD。

## S1：新产品或新功能探索

适合：要做新产品、新模块、新 AI 能力，但还没证明用户真的需要。

直接这样说：

```text
我们想做一个面向销售团队的 AI 客户跟进助手，目前只是想法。请按新产品探索处理，帮我梳理目标用户、核心场景、用户问题、关键假设和验证计划。
```

会得到：探索 brief、目标用户、问题假设、机会树、验证计划。

先别继续：目标用户写成“所有人”、问题只写“提升效率”、没有验证计划时，不要进入 PRD。

## S2：已有产品优化

适合：已有功能效果不好，用户投诉，使用率、转化率、留存或满意度低。

直接这样说：

```text
导出功能上线后使用率很低，用户也反馈路径难找。请先帮我诊断原因，区分数据问题、体验问题和功能价值问题，再给优化方向。
```

会得到：现状诊断、问题假设、指标基线、优化方案、实验计划。

先别继续：没有基线指标、没有用户反馈、问题路径不清楚时，不要直接改版。

## S3：用户反馈转路线图

适合：客户反馈、销售需求、客服工单很多，需要整理成需求池和路线图候选。

直接这样说：

```text
这里有 30 条客户反馈，请帮我按用户问题聚类，区分事实、观点和假设，并整理成路线图候选项。
```

会得到：反馈主题、问题聚类、证据表、需求候选、优先级建议。

先别继续：只知道“客户要什么”，不知道“为什么要”时，不要直接排期。

## S4：PRD 标准化

适合：需求已经准备交给设计、研发或测试，需要写清楚范围、规则、验收和边界。

直接这样说：

```text
请把这个需求整理成可评审 PRD。按需求类型自适应，不适用的模块请说明原因，并检查是否能交给 UI 和研发。
```

会得到：自适应 PRD、用户故事、业务规则、验收标准、风险和交接检查。

先别继续：没有范围、非目标、验收标准、异常状态时，不要交给研发。

## S5：用户研究

适合：要访谈用户、验证假设、理解流失原因，或总结访谈记录。

直接这样说：

```text
我想访谈 8 个新用户，了解他们为什么没有完成首次配置。请帮我设计研究目标、招募标准、访谈问题和访谈后的洞察整理方式。
```

会得到：研究目标、招募标准、访谈脚本、记录方式、洞察总结。

先别继续：访谈问题带引导、只问“你想不想要这个功能”时，需要重写访谈方案。

## S6：优先级和路线图

适合：需求很多、资源有限，需要决定先做什么、后做什么、为什么不做。

直接这样说：

```text
我们有 12 个候选需求，请帮我按用户价值、业务价值、成本、风险和依赖做优先级判断，并输出一版路线图建议。
```

会得到：优先级判断、取舍理由、路线图、暂缓/拒绝说明。

先别继续：业务目标不清、需求粒度不一致、只按谁声音大排序时，不要定路线图。

## S7：指标和实验

适合：要定义指标、埋点、实验方案，或判断一次改动是否有效。

直接这样说：

```text
我们要优化新用户首次配置流程，请帮我定义核心指标、过程指标、护栏指标、埋点事件和实验判断标准。
```

会得到：指标树、埋点口径、实验方案、成功/失败判断标准。

先别继续：没有基线、没有护栏指标、样本不足却要求确定结论时，不要做强判断。

## S8：战略和商业模式

适合：判断一个方向是否值得做，分析市场、竞品、定位、商业模式。

直接这样说：

```text
我们在考虑进入企业知识库 AI 助手方向。请从目标客户、市场机会、竞品替代、差异化定位、商业模式和关键风险做分析。
```

会得到：目标市场、客户细分、竞品分析、定位假设、商业模式、战略风险。

先别继续：市场定义太大、只做功能对比、没有说明为什么我们能赢时，不要下结论。

## S9：上线准备和 GTM

适合：功能快上线，要检查发布范围、灰度、监控、回滚、销售客服准备。

直接这样说：

```text
这个功能下周上线，请帮我检查上线准备：发布范围、灰度策略、监控指标、回滚方案、销售客服话术和上线后复盘计划。
```

会得到：上线检查表、GTM 准备、风险清单、监控和复盘计划。

先别继续：没有灰度、监控、回滚、客服说明时，不要正式上线。

## S10：PM 日常运营协作

适合：整理会议纪要、行动项、复盘、sprint plan、OKR、风险和开放问题。

直接这样说：

```text
请把这次需求评审会议整理成 PM 日常运营产物：结论、关键决策、行动项、风险、开放问题和下次跟进事项。
```

会得到：会议结论、决策记录、行动项、风险、开放问题。

先别继续：行动项没有负责人、决策没有理由、风险没人跟进时，需要补齐。

## 与 UI 和研发的衔接

交给 UI 前，至少说清：目标用户、使用场景、核心任务、用户路径、页面状态、异常状态、权限差异、关键文案。

交给研发前，至少说清：需求范围、非目标、功能规则、数据规则、权限规则、异常规则、验收标准、埋点口径、上线和回滚方案。

<callout emoji="✅" background-color="light-green" border-color="green">
  <p><b>交接前检查：</b>如果你不能清楚回答“谁在什么场景下完成什么任务”“哪些规则不能变”“怎么验收算完成”，就还不适合交给 UI 或研发。</p>
</callout>

## 常见问题

### 我需要记住场景编号吗？

不需要。日常使用时，直接描述真实工作内容即可。

### 为什么它有时不直接写 PRD？

因为很多任务还处在澄清、探索、研究、排优先级或定义指标阶段。过早写 PRD 会把模糊问题传给下游。

### 提示 BLOCKED 怎么办？

看缺口，补信息。常见缺口是目标用户、问题证据、业务目标、范围、验收标准、指标或风险。

### 我不知道怎么描述怎么办？

直接说：

```text
我不知道这件事该怎么推进，请先帮我判断场景，并告诉我需要补哪些信息。
```
