# PM Superpowers 产品经理工作流助手：安装与场景使用指南

这份文档给内部产品经理使用。每个要使用 PM Superpowers 的同事，都需要先完成一次安装；团队通知升级时，再按步骤更新。

安装和更新不需要理解技术原理，复制命令执行即可。装好以后，你只需要描述真实业务背景，PM Superpowers 会判断产品场景，并带你走对应工作流。

<callout emoji="📌" background-color="light-blue" border-color="blue">
  <p><b>先说结论：</b>先安装，再使用；先判断场景，再写产物；信息不清楚时，先补信息，不要硬往下走。</p>
</callout>

## 安装插件

复制下面两行即可完成安装；插件地址：`https://github.com/linbiqiu/pm-superpowers`。

```bash
codex plugin marketplace add linbiqiu/pm-superpowers --ref main
codex plugin add pm-superpowers@pm-superpowers-internal
```

安装后新开一个 Codex 对话窗口。旧窗口通常不会自动加载新能力。

## 如何更新

收到团队更新通知时，复制下面两行即可完成更新。

```bash
codex plugin marketplace upgrade pm-superpowers-internal
codex plugin add pm-superpowers@pm-superpowers-internal
```

更新后，新开一个 Codex 对话窗口。

## 它帮你做什么

PM Superpowers 是产品经理工作流助手，不是单纯的 PRD 生成器。

它会帮你：

- 判断当前属于哪个产品场景。
- 追问缺失信息。
- 输出中文产品产物。
- 检查是否能交给 UI、研发、测试或运营。
- 信息不足时提示 `BLOCKED`，避免把模糊需求传给下游。

## 第一次怎么开始

不要一上来就说“帮我写 PRD”。先让它判断场景。

可以这样说：

```text
我现在有一个产品需求，但还不确定应该怎么推进。请先帮我判断属于哪个产品场景，并告诉我继续前缺哪些信息。
```

如果已经有业务背景，就说清楚：

```text
我们想做一个给销售主管看的 AI 周报功能，目标是减少人工整理时间。请先帮我判断应该走哪个产品工作流，不要直接写 PRD。
```

## 开始前先回答五个问题

| 问题 | 为什么重要 |
| --- | --- |
| 这件事从哪里来？ | 判断是反馈、指标、老板想法、上线任务，还是战略方向 |
| 服务谁？ | 避免做出没人真正需要的需求 |
| 现在要做什么决策？ | 决定做不做、先做哪个、怎么做、能不能上线 |
| 最大不确定是什么？ | 找到需要先补的信息 |
| 产物给谁用？ | 决定输出给自己、设计、研发、销售客服还是管理层 |

<callout emoji="📌" background-color="light-yellow" border-color="yellow">
  <p><b>判断规则：</b>这五个问题里有两个以上答不清，就先做澄清，不要直接写 PRD。</p>
</callout>

## S0：模糊需求澄清

适合：你手里只有一句想法、一个老板要求、一条客户诉求，或者一句“竞品都有了，我们也要做”，但还说不清用户是谁、问题是什么、为什么现在要做。

典型例子：

- “老板说我们也要做 AI 助手”，但没人说清楚给谁用、解决什么问题。
- “客户说想要自动报表”，但不知道是想省时间、减少错误，还是满足管理层查看。
- “竞品上线了某个功能”，但还没判断它对我们的用户是否重要。

它会怎么带你走：

- 先把一句话拆成背景、目标用户、用户问题、业务目标、已有证据。
- 再判断这件事更像新功能探索、已有功能优化、反馈整理，还是 PRD 准备。
- 最后列出必须补齐的信息，并告诉你下一步应该进入哪个场景。

会用到的产品方法：需求入口分流、关键信息补全、假设识别、门禁检查。

使用例子：

```text
老板提到我们也要做 AI 助手，但目标用户和具体场景还不清楚。请先帮我做需求澄清，判断这属于哪个产品场景，并列出必须补齐的信息。
```

会得到：澄清后的任务说明、已知信息、缺失信息、关键假设、阻断项、推荐工作流。

进入下一步门槛：至少能说清“谁在什么场景下遇到什么问题”“为什么现在要处理”“下一步要做什么决策”。说不清时，不要写 PRD。

## S1：新产品或新功能探索

适合：你要做一个新产品、新模块、新 AI 能力，或者从 0 到 1 尝试一个方向，但还没有证明用户真的需要，也没有确定最小可验证版本。

典型例子：

- 想做“销售 AI 客户跟进助手”，但还不知道销售最痛的是写跟进记录、找客户线索，还是提醒下一步动作。
- 想给后台加“智能分析模块”，但还没有确认哪些岗位每天真的需要看分析结论。
- 想做一个全新的会员体系，但还没搞清用户愿意为什么价值付费。

它会怎么带你走：

- 先定义目标用户和具体场景，避免把用户写成“所有人”。
- 再拆出用户问题、现有替代方案、关键假设和风险。
- 然后设计验证方式，比如访谈、原型测试、小流量试点、MVP。

会用到的产品方法：用户画像、价值主张、机会解决方案树、假设验证、MVP 设计。

使用例子：

```text
我们想做一个面向销售团队的 AI 客户跟进助手，目前只是想法。请按新产品探索处理，帮我梳理目标用户、核心场景、用户问题、关键假设和验证计划。
```

会得到：探索 brief、目标用户、核心场景、问题假设、机会树、验证计划、MVP 建议。

进入下一步门槛：目标用户不能是“所有人”，用户问题不能只写“提升效率”，至少要有一个可验证假设和验证计划。没有验证计划时，不要进入 PRD。

## S2：已有产品优化

适合：功能已经上线，但效果不好、用户投诉、流程难用、转化低、留存低、使用率低，或者团队感觉“这个功能应该改一版”。

典型例子：

- 导出功能使用率低，用户反馈“找不到入口”。
- 新用户配置流程流失高，但不知道是步骤太多、文案看不懂，还是权限卡住了。
- 搜索功能被投诉“不准”，但还没区分是召回问题、排序问题，还是用户关键词表达不一致。

它会怎么带你走：

- 先确认现状：功能目标、当前指标、用户反馈、关键路径。
- 再拆原因：价值不够、入口不明显、流程太长、规则不清、性能体验差。
- 最后给优化方案，并建议用实验或灰度验证。

会用到的产品方法：用户旅程、问题归因、指标基线、功能优先级、实验设计。

使用例子：

```text
导出功能上线后使用率很低，用户也反馈路径难找。请先帮我诊断原因，区分数据问题、体验问题和功能价值问题，再给优化方向。
```

会得到：现状诊断、问题假设、指标基线、优化方案、实验计划、改版风险。

进入下一步门槛：至少要有一个基线指标或真实反馈，并能说清问题发生在哪条路径。没有证据时，不要直接改版。

## S3：用户反馈转路线图

适合：客户、销售、客服、实施、运营不断提需求，信息很多但很乱，需要从“谁说了什么”整理成“哪些问题值得进入路线图”。

典型例子：

- 销售收集了几十条客户要求，有的是个性化定制，有的是普遍痛点。
- 客服工单里反复出现同类问题，但没有人整理成产品机会。
- 大客户要求插队做功能，团队需要判断是否值得影响当前排期。

它会怎么带你走：

- 先把反馈按用户问题聚类，而不是按客户原话照抄。
- 再区分事实、观点、解决方案诉求和我们自己的假设。
- 最后形成需求候选池，并给出优先级、证据强度和路线图建议。

会用到的产品方法：反馈聚类、证据分级、需求池、优先级评估、路线图候选。

使用例子：

```text
这里有 30 条客户反馈，请帮我按用户问题聚类，区分事实、观点和假设，并整理成路线图候选项。
```

会得到：反馈主题、问题聚类、证据表、需求候选、优先级建议、需要二次确认的问题。

进入下一步门槛：不能只知道“客户要什么”，还要知道“为什么要”“多少客户有类似问题”“不做会怎样”。不知道原因时，不要直接排期。

## S4：PRD 标准化

适合：需求已经比较清楚，准备进入设计、研发、测试评审，需要把范围、规则、状态、验收、埋点和交接信息写清楚。

典型例子：

- 已确定要做“AI 周报生成”，需要交给 UI 设计页面和交互。
- 已确定要改“审批流规则”，需要研发明确权限、状态流转和异常处理。
- 已确定要上线“导出报表”，需要测试知道哪些场景必须验收。

它会怎么带你走：

- 先判断需求类型，不强行套一个固定 PRD 模板。
- 再补齐目标用户、使用场景、功能范围、非目标、业务规则、异常状态。
- 最后检查是否能交给 UI、研发、测试，不足时会提示阻断项。

会用到的产品方法：PRD、用户故事、业务规则、测试场景、下游交接检查。

使用例子：

```text
请把这个需求整理成可评审 PRD。按需求类型自适应，不适用的模块请说明原因，并检查是否能交给 UI 和研发。
```

会得到：自适应 PRD、用户故事、业务规则、页面状态、异常场景、验收标准、风险和交接检查。

进入下一步门槛：必须说清需求范围、非目标、关键规则、验收标准和异常状态。缺这些时，不要交给研发。

## S5：用户研究

适合：你不知道用户真实想法，需要访谈、调研、可用性测试、总结访谈记录，或者验证某个产品假设是否成立。

典型例子：

- 新用户没有完成首次配置，需要了解他们卡在哪一步。
- 客户说“报表不好用”，需要访谈不同角色，弄清楚具体不好用在哪里。
- 团队想做新功能，但需要先验证用户是否真的有这个问题。

它会怎么带你走：

- 先明确研究目标和要验证的假设，避免泛泛地“了解用户”。
- 再设计招募标准、访谈问题、记录模板和洞察整理方式。
- 访谈后可以继续把原始记录整理成主题、证据、洞察和产品机会。

会用到的产品方法：访谈脚本、用户画像、用户旅程、访谈总结、需求洞察。

使用例子：

```text
我想访谈 8 个新用户，了解他们为什么没有完成首次配置。请帮我设计研究目标、招募标准、访谈问题和访谈后的洞察整理方式。
```

会得到：研究目标、招募标准、访谈脚本、记录模板、洞察总结框架、后续产品机会。

进入下一步门槛：访谈问题不能诱导用户回答，不能只问“你想不想要这个功能”。如果问题带答案，需要先重写访谈方案。

## S6：优先级和路线图

适合：候选需求很多、资源有限、各方都觉得自己最急，需要决定先做什么、后做什么、暂缓什么，以及怎么解释取舍。

典型例子：

- 一个季度只有 2 个研发，但需求池里有 15 个需求。
- 销售、运营、老板都在催不同事项，需要统一排序依据。
- 产品线要规划下个季度路线图，需要平衡增长、体验、技术债和大客户承诺。

它会怎么带你走：

- 先统一需求粒度，避免把“大方向”和“小优化”放在一起比。
- 再按用户价值、业务价值、成本、风险、依赖、时效性评估。
- 最后输出路线图建议，并写清楚为什么做、为什么不做、什么时候再看。

会用到的产品方法：优先级框架、路线图、干系人地图、机会成本、取舍说明。

使用例子：

```text
我们有 12 个候选需求，请帮我按用户价值、业务价值、成本、风险和依赖做优先级判断，并输出一版路线图建议。
```

会得到：优先级判断、取舍理由、路线图建议、暂缓/拒绝说明、需要补充的数据。

进入下一步门槛：业务目标要明确，需求粒度要可比较，不能只按谁声音大排序。目标不清时，不要定路线图。

## S7：指标和实验

适合：你要定义指标、埋点、实验方案、看板口径，或者判断一次产品改动到底有没有效果。

典型例子：

- 想优化新用户配置流程，但不知道核心指标应该看完成率、耗时，还是激活率。
- 改了推荐策略，需要判断点击率提高是否真的带来业务价值。
- 准备做 A/B 实验，但不知道样本量、成功标准和护栏指标怎么定。

它会怎么带你走：

- 先明确产品目标，再拆核心指标、过程指标、结果指标和护栏指标。
- 再定义埋点事件、口径、分群和分析维度。
- 最后设计实验或观察方案，明确成功、失败和无法判断的标准。

会用到的产品方法：北极星指标、指标看板、A/B 测试、队列分析、实验设计。

使用例子：

```text
我们要优化新用户首次配置流程，请帮我定义核心指标、过程指标、护栏指标、埋点事件和实验判断标准。
```

会得到：指标树、埋点口径、实验方案、成功/失败判断标准、数据风险提示。

进入下一步门槛：要有基线、口径、护栏指标和样本判断。样本不足或口径不一致时，不要做强结论。

## S8：战略和商业模式

适合：你要判断一个方向是否值得做，或者需要分析市场、目标客户、竞品、定位、商业模式、价格和进入策略。

典型例子：

- 公司考虑进入“企业知识库 AI 助手”方向，需要判断机会和风险。
- 现有产品想从中小客户转向大客户，需要评估 ICP、销售动作和产品能力差距。
- 竞品开始降价，需要判断我们是跟进、差异化，还是调整套餐。

它会怎么带你走：

- 先定义市场和客户，不把市场写成一个过大的概念。
- 再分析客户痛点、替代方案、竞争格局、差异化和商业模式。
- 最后形成战略判断：值得做什么、不做什么、先验证什么。

会用到的产品方法：市场细分、ICP、竞品分析、SWOT、PESTLE、五力模型、定位、定价。

使用例子：

```text
我们在考虑进入企业知识库 AI 助手方向。请从目标客户、市场机会、竞品替代、差异化定位、商业模式和关键风险做分析。
```

会得到：目标市场、客户细分、竞品分析、定位假设、商业模式、定价思路、战略风险。

进入下一步门槛：市场定义不能太空，竞品分析不能只列功能，必须说明“为什么我们能赢”。说不清时，不要下战略结论。

## S9：上线准备和 GTM

适合：功能快上线，需要检查发布范围、灰度策略、监控指标、回滚方案、销售客服话术、帮助文档和上线后复盘。

典型例子：

- 新功能下周上线，但还没确定哪些客户先开、出了问题怎么回滚。
- ToB 功能要对外发布，销售需要知道怎么介绍，客服需要知道怎么答疑。
- 灰度上线后要观察效果，但还没定义监控指标和复盘时间。

它会怎么带你走：

- 先检查上线范围、灰度批次、风险点和回滚条件。
- 再补齐内外部沟通材料，比如发布说明、销售话术、客服 FAQ。
- 最后制定上线后监控和复盘计划，避免上线后没人看结果。

会用到的产品方法：上线检查、GTM、发布说明、风险清单、上线后复盘。

使用例子：

```text
这个功能下周上线，请帮我检查上线准备：发布范围、灰度策略、监控指标、回滚方案、销售客服话术和上线后复盘计划。
```

会得到：上线检查表、GTM 准备、风险清单、监控指标、回滚方案、销售客服材料、复盘计划。

进入下一步门槛：必须有灰度、监控、回滚、客服说明和复盘安排。缺这些时，不要正式上线。

## S10：PM 日常运营协作

适合：产品经理日常要整理会议、同步进展、拆 sprint、复盘问题、管理行动项、更新 OKR、记录关键决策。

典型例子：

- 需求评审开完了，需要把结论、争议、行动项整理清楚。
- Sprint 计划要确定本周做什么、风险是什么、哪些依赖没解决。
- 项目复盘要区分事实、原因、改进动作，而不是只写“下次注意”。

它会怎么带你走：

- 先提炼结论和关键决策，避免会议纪要只是流水账。
- 再列行动项、负责人、截止时间、风险和开放问题。
- 最后把关键决策沉淀到项目记录里，方便后续追溯。

会用到的产品方法：会议总结、决策记录、行动项管理、Sprint Plan、复盘、OKR。

使用例子：

```text
请把这次需求评审会议整理成 PM 日常运营产物：结论、关键决策、行动项、风险、开放问题和下次跟进事项。
```

会得到：会议结论、决策记录、行动项、风险、开放问题、下次跟进事项。

进入下一步门槛：行动项必须有负责人和时间，决策必须有理由，风险必须有人跟进。缺这些时，会议产物不算完成。

## 与 UI 和研发的衔接

交给 UI 前，至少说清：目标用户、使用场景、核心任务、用户路径、页面状态、异常状态、权限差异、关键文案。

交给研发前，至少说清：需求范围、非目标、功能规则、数据规则、权限规则、异常规则、验收标准、埋点口径、上线和回滚方案。

## 常见问题

### 我需要记住场景编号吗？

不需要。日常使用时，直接描述真实工作内容即可。

### 为什么它有时不直接写 PRD？

因为很多任务还处在澄清、探索、研究、排优先级或定义指标阶段。过早写 PRD 会把模糊问题传给下游。

### 提示 BLOCKED 怎么办？

看缺口，补信息。常见缺口是目标用户、问题证据、业务目标、范围、验收标准、指标或风险。

### 我不知道怎么描述怎么办？

直接说：

```text
我不知道这件事该怎么推进，请先帮我判断场景，并告诉我需要补哪些信息。
```
