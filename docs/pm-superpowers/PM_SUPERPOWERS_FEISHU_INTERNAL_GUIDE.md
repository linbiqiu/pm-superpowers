# PM Superpowers 产品经理工作流插件：安装、使用与高频场景指南

这份文档给内部产品经理、新接触产品工作的同事、业务转产品/项目转产品的同事使用。你不需要先理解所有产品方法论，也不需要手动挑选底层技能。你只需要把真实业务背景说清楚，PM Superpowers 会先判断场景，再带你走对应的产品工作流。

## 一句话说明

PM Superpowers 是一个 Codex 产品经理工作流插件。它不是普通软件，也不是一个固定提示词，而是一套面向产品工作的场景化工作流系统。

安装后，你可以在 Codex 里直接描述产品工作，例如：

```text
我们想做一个给销售主管看的 AI 周报功能，目标是减少人工整理时间。请帮我按 PM Superpowers 的方式梳理产品思路。
```

插件会做三件事：

- 先判断你当前属于哪类产品场景。
- 再选择合适的产品工作流和底层 PM 方法技能。
- 最后按中文规范输出可评审、可交接、可沉淀的产品产物。

它的目标不是帮你“快速写一份看起来完整的 PRD”，而是帮你把产品问题想清楚、把证据和假设分清楚、把产物做到可以交给 UI、研发、测试、运营或管理层继续使用。

## 为什么团队需要这个插件

产品工作里最常见的问题，不是大家不会写文档，而是经常在不合适的阶段写了不合适的文档。

比如：

- 只有一句老板想法，就开始写 PRD。
- 客户提出一个功能，就直接照单全收。
- 用户说页面不好用，就马上要求设计改版。
- 指标下降了，但没有定义基线和判断口径。
- 需求文档很长，但研发看完仍然不知道边界、验收和非目标。
- 功能做完了，却没有上线检查、监控指标、回滚方案和 GTM 准备。

PM Superpowers 要解决的是这些场景判断和工作流规范问题。它会把产品工作拆成可学习、可执行、可检查的场景流程，让新人知道下一步该做什么，也让资深产品经理有统一的团队标准。

## 适合谁使用

- 刚接触产品工作的同事：用它理解产品工作不是从 PRD 开始，而是从场景判断开始。
- 日常负责需求、迭代、用户反馈、上线的产品经理：用它提升需求质量和交接质量。
- 需要统一团队规范的产品负责人：用它沉淀团队的产品工作流、模板和门禁规则。
- 需要和 UI/研发智能体协作的同事：用它把产品产物整理到可下游交接的状态。

## 安装前准备

请确认你已经具备：

- 已安装 Codex。
- 终端里可以运行 `codex` 命令。
- 能访问插件仓库：`https://github.com/linbiqiu/pm-superpowers`。

可以先运行：

```bash
codex plugin --help
```

如果能看到帮助信息，说明 Codex CLI 可用。

## 推荐安装方式

在终端执行：

```bash
codex plugin marketplace add linbiqiu/pm-superpowers --ref main
codex plugin add pm-superpowers@pm-superpowers-internal
```

安装完成后检查：

```bash
codex plugin list
```

看到类似下面的信息，就说明插件已经安装并启用：

```text
pm-superpowers@pm-superpowers-internal  installed, enabled
```

重要提醒：安装或更新插件后，请新开一个 Codex thread。已经打开的旧 thread 通常不会自动加载新安装或新更新的插件技能。

## 更新插件

Codex 插件不会在后台静默自动更新。原因是 PM Superpowers 会影响团队工作流、模板、门禁和交接标准，更新应该可感知、可追踪。

如果你已经下载了插件仓库，可以在仓库根目录执行：

```bash
scripts/check_pm_superpowers_update.sh
scripts/update_pm_superpowers.sh
```

如果没有本地仓库，也可以手动执行：

```bash
codex plugin marketplace upgrade pm-superpowers-internal
codex plugin add pm-superpowers@pm-superpowers-internal
```

更新完成后，同样需要新开一个 Codex thread。

## 第一次怎么用

第一次使用时，不建议一上来就说“帮我写 PRD”。更好的方式是先让插件判断场景。

推荐这样说：

```text
我现在有一个比较粗的产品想法，请先帮我判断应该走哪个 PM 工作流，并告诉我继续推进前还缺哪些关键信息。
```

如果你已经有真实业务背景，可以这样说：

```text
我们想做一个给客服主管看的质检看板，目前只有初步想法。请按 PM Superpowers 的方式先做场景判断，不要直接写 PRD。
```

如果你已经有一份 PRD，可以这样说：

```text
请按 PM Superpowers 检查这份 PRD 是否可以交给 UI 和研发。如果不能，请指出阻断项、风险和必须补齐的信息。
```

如果你希望把一个产品项目长期管理起来，可以这样说：

```text
请用 PM Superpowers 初始化当前项目的产品工作区，创建 AGENTS.md、产品上下文、决策记录、证据表、风险清单和下游交接检查。
```

## 使用时最重要的五个问题

每次打开插件前，先用人话回答这五个问题。答得越清楚，插件越能直接进入正确工作流；答不清楚时，插件会先帮你澄清。

| 问题 | 通俗解释 | 答不清会怎样 |
| --- | --- | --- |
| 这件事从哪里来？ | 是老板想法、客户反馈、指标异常、会议结论，还是上线任务？ | 插件无法判断场景 |
| 这件事服务谁？ | 谁是用户、客户、内部角色或下游团队？ | 容易写出没人真正需要的需求 |
| 你想做成什么决策？ | 是决定做不做、先做哪个、怎么做、能不能上线，还是怎么复盘？ | 产物会变成泛泛分析 |
| 现在最不确定的是什么？ | 用户、问题、证据、方案、范围、指标、风险，哪个最不清楚？ | 容易跳过关键思考 |
| 下游是谁？ | 给自己继续探索、给设计、给研发、给销售客服，还是给管理层？ | 输出格式和深度会错 |

如果这五个问题里有两个以上答不清，不要直接进入 PRD。先让插件做模糊需求澄清。

## 插件的核心原则

- 场景优先：先判断当前是什么产品场景，再选择工作流。
- 中文优先：默认输出中文产品产物，英文框架名会保留并解释。
- 证据优先：把事实、用户原话、数据、假设、意见和承诺分开。
- 门禁优先：信息不足时输出 `BLOCKED`，阻止没想清楚就进入下一步。
- 交接优先：产品产物最终要能交给 UI、研发、测试、运营或管理层使用。
- 沉淀优先：关键决策、证据、风险和开放问题要可追踪，不只停留在一次对话里。

## 如何理解 BLOCKED

`BLOCKED` 不是插件失败，而是在提醒你：现在的信息还不足以支撑下一步。

以下情况就应该阻断：

- 不知道用户是谁。
- 不知道问题是什么。
- 不知道当前要支持什么决策。
- 没有证据，却要求给确定结论。
- 没有范围和验收标准，却要交给研发。
- 没有关键状态和用户流，却要交给 UI。
- 没有监控和回滚方案，却要上线。

好的产品经理不是永远往前推，而是知道什么时候必须停下来补信息。

## 高频场景总览

| 编号 | 场景 | 典型触发 | 最终产物 |
| --- | --- | --- | --- |
| S0 | 模糊需求澄清 | 只有一句想法，不知道怎么推进 | 澄清 brief、推荐场景、缺口清单 |
| S1 | 新产品或新功能探索 | 想做新产品、新模块、新 AI 能力 | 探索 brief、假设地图、验证计划 |
| S2 | 已有产品优化 | 指标不好、用户投诉、流程难用 | 现状诊断、优化方案、实验计划 |
| S3 | 反馈转路线图 | 客户反馈、销售需求、客服工单很多 | 反馈主题、需求池、路线图候选 |
| S4 | PRD 标准化 | 准备交给 UI 或研发 | 自适应 PRD、验收标准、交接检查 |
| S5 | 用户研究 | 要访谈用户或总结访谈 | 研究计划、访谈脚本、洞察总结 |
| S6 | 优先级和路线图 | 很多需求不知道先做哪个 | 优先级判断、路线图、取舍说明 |
| S7 | 指标和实验 | 不知道看什么指标或怎么实验 | 指标树、实验方案、判断标准 |
| S8 | 战略和商业模式 | 要判断方向、市场、定位、商业模式 | 战略分析、定位、商业模式假设 |
| S9 | 上线准备和 GTM | 功能快上线或准备发布 | 上线检查、GTM 计划、风险清单 |
| S10 | PM 日常运营协作 | 会议、复盘、sprint、OKR、行动项 | 会议纪要、行动项、决策记录 |

## S0：模糊需求澄清

### 你可能正遇到的情况

别人给你的输入很粗：老板说要做 AI 助手，客户说想要报表，销售说某功能很重要，竞品有一个新能力，或者团队只说“帮我看看这个产品怎么做”。

这些都不是可以直接写 PRD 的需求，只是产品工作的入口。

### 这个场景要解决什么

S0 的目标不是马上给答案，而是把模糊输入变成清楚的产品任务。插件会帮你判断这件事到底应该进入新产品探索、已有产品优化、反馈转路线图、PRD 标准化，还是其他场景。

### 你应该怎么用

```text
我现在只有一个比较模糊的产品想法，还不知道该走什么流程。请先按模糊需求澄清处理，帮我判断这是哪个产品场景，并列出继续推进前必须补齐的信息。
```

### 合格产物应该包含

- 当前判断：这是什么场景，不是什么场景。
- 已知信息：背景、来源、目标用户、业务目标。
- 假设清单：哪些只是猜测，尚未被验证。
- 阻断项：继续推进前必须补齐什么。
- 下一步：推荐进入哪个工作流。

## S1：新产品或新功能探索

### 你可能正遇到的情况

团队想做一个新产品、新模块、新 AI 能力、新商业化能力或新用户场景。想法看起来有价值，但还不知道用户是不是真的需要。

### 这个场景要解决什么

S1 不是把想法写成需求，而是证明这个想法是否值得继续。重点是把热情变成假设，把假设变成验证计划。

### 你应该怎么用

```text
我们想做一个面向销售团队的 AI 客户跟进助手，目前还只是想法。请按新产品探索场景处理：先帮我明确目标用户、核心场景、用户问题、关键假设和验证计划，不要直接写 PRD。
```

### 工作流会怎么走

1. 明确目标用户和使用场景。
2. 把用户问题和业务目标分开。
3. 梳理现有替代方案。
4. 建立机会、方案、实验之间的关系。
5. 找出最高风险假设。
6. 设计低成本验证。
7. 通过门禁后再进入 PRD 或 UI。

### 什么时候不能继续

- 目标用户只写“所有人”或“所有销售”。
- 问题只写“提升效率”，没有具体场景。
- 没有替代方案分析。
- 没有关键假设排序。
- MVP 只是功能合集，看不出要验证什么。

## S2：已有产品优化

### 你可能正遇到的情况

已有功能上线了，但转化率、留存、使用率、满意度或投诉情况不理想。用户说难用，业务说效果不好，团队想做一轮优化。

### 这个场景要解决什么

S2 的核心不是“马上改版”，而是先诊断问题：到底是用户路径问题、信息架构问题、功能价值问题、数据口径问题，还是运营触达问题。

### 你应该怎么用

```text
导出功能使用率很低，用户也经常投诉。请按已有产品优化场景帮我诊断原因，先不要直接给改版方案。
```

### 工作流会怎么走

1. 明确要优化的功能和用户路径。
2. 确认当前症状和基线指标。
3. 汇总用户反馈、数据和业务约束。
4. 诊断可能原因。
5. 生成优化方案。
6. 设计实验或验证方式。
7. 输出优先级和风险。

### 什么时候不能继续

- 没有当前指标或用户反馈。
- 不知道问题发生在哪个路径。
- 把“用户说难用”直接等同于“必须重做 UI”。
- 没有判断优化是否有效的指标。

## S3：用户反馈转路线图

### 你可能正遇到的情况

你手上有很多客户反馈、销售需求、客服工单、老板要求或竞品参考。大家都说重要，但资源有限，必须决定哪些进入路线图。

### 这个场景要解决什么

S3 的重点是把杂乱输入变成可决策的需求池。客户提出的功能方案不等于真实问题，必须先还原用户痛点，再决定是否进入路线图。

### 你应该怎么用

```text
这里有 30 条客户反馈，请帮我按用户问题聚类，区分证据和观点，并转成路线图候选项。
```

### 工作流会怎么走

1. 标准化每条反馈的来源、用户类型、场景和原话。
2. 区分事实、用户原话、信号、假设和意见。
3. 把相似反馈聚类成主题。
4. 把功能请求还原成用户问题。
5. 按影响、频率、业务价值、风险和成本排序。
6. 形成路线图候选。
7. 记录选择、延期和拒绝的理由。

### 什么时候不能继续

- 反馈没有来源。
- 只记录“客户要什么”，没有记录“为什么要”。
- 只按客户级别或声音大小排序。
- 没有说明哪些需求不做以及原因。

## S4：PRD 标准化

### 你可能正遇到的情况

你已经准备把需求交给设计、研发或测试，或者已有 PRD 结构混乱，需要重写、补齐和评审。

### 这个场景要解决什么

S4 的目标不是把 PRD 写得越长越好，而是让下游不用猜产品决策。PRD 要清楚说明用户、问题、目标、范围、流程、规则、边界、验收、指标和非目标。

### 你应该怎么用

```text
请帮我把这个需求整理成可以评审的 PRD。注意按需求类型自适应，不适用的模块请标注不适用原因，不要硬填。
```

### 工作流会怎么走

1. 判断 PRD 类型：轻量需求、标准功能、UI/体验、后台/接口/数据、优化实验、上线运营等。
2. 检查问题、用户、证据、目标、范围和指标。
3. 选择适用模块，不适用模块标注原因。
4. 补齐用户故事、系统场景或 job stories。
5. 补齐验收标准、边界状态、异常状态和非目标。
6. 做门禁检查。
7. 做 UI/研发下游 readiness 检查。

### 什么时候不能继续

- 只写了解决方案，没有写用户问题。
- 没有范围和非目标。
- 没有验收标准。
- 没有边界状态、异常状态和权限规则。
- 交给 UI 但没有关键状态和用户流。
- 交给研发但没有数据、接口、规则或验收口径。

## S5：用户研究

### 你可能正遇到的情况

你要访谈用户、验证需求、理解流失原因、总结访谈记录，或者把定性研究转成产品决策。

### 这个场景要解决什么

S5 的重点不是“问用户想要什么功能”，而是理解用户在真实场景下的行为、动机、阻碍和替代方案。

### 你应该怎么用

```text
我想访谈 8 个新用户，验证他们为什么没有完成首次配置。请帮我设计研究目标、招募标准、访谈脚本和访谈后的洞察整理方式。
```

### 工作流会怎么走

1. 明确研究目标和要支持的产品决策。
2. 定义目标用户和招募标准。
3. 设计非引导式访谈问题。
4. 规划记录方式和标签体系。
5. 总结用户原话、行为证据和洞察。
6. 把洞察转成机会、假设或需求候选。

### 什么时候不能继续

- 访谈问题引导用户回答。
- 只问“你想不想要这个功能”。
- 没有招募标准。
- 访谈总结只有观点，没有用户原话和证据。
- 洞察没有连接到产品决策。

## S6：优先级和路线图

### 你可能正遇到的情况

需求很多，资源有限。业务、客户、研发、老板都有诉求，你需要决定先做什么、后做什么、为什么不做某些事。

### 这个场景要解决什么

S6 的重点是建立透明的取舍逻辑，而不是找一个看起来客观的打分表。好的路线图应该围绕结果，而不是堆功能。

### 你应该怎么用

```text
我们现在有 12 个候选需求，请帮我建立优先级判断，并输出 outcome roadmap。请说明每个取舍背后的证据、假设和风险。
```

### 工作流会怎么走

1. 明确业务目标和产品结果。
2. 统一候选需求的描述格式。
3. 识别每个需求的用户价值、业务价值、风险、成本和依赖。
4. 选择合适的优先级框架。
5. 输出路线图和取舍说明。
6. 记录关键决策。

### 什么时候不能继续

- 没有业务目标。
- 候选需求粒度不一致。
- 只按老板优先级或客户级别排序。
- 没有说明为什么暂缓或拒绝。
- 路线图只有功能，没有结果指标。

## S7：指标和实验

### 你可能正遇到的情况

你要定义北极星指标、功能指标、埋点方案、实验方案，或者判断一次改动是否真的有效。

### 这个场景要解决什么

S7 的核心是让团队知道“我们到底用什么判断好坏”。不是所有事情都能 A/B 测试，但每个重要产品动作都应该有判断标准。

### 你应该怎么用

```text
我们要优化新用户首次配置流程，请帮我定义指标树、关键事件、实验方案和上线后的判断标准。
```

### 工作流会怎么走

1. 明确产品目标和用户行为。
2. 定义北极星指标或核心结果指标。
3. 拆解输入指标、过程指标和护栏指标。
4. 设计埋点和数据口径。
5. 选择实验方式。
6. 设定成功、失败和继续观察的判断标准。

### 什么时候不能继续

- 指标只写“提升体验”。
- 没有基线。
- 没有护栏指标。
- 实验样本不足却要求确定结论。
- 埋点事件没有定义触发时机和属性。

## S8：战略和商业模式

### 你可能正遇到的情况

你需要判断一个方向是否值得做，分析市场、竞品、定位、商业模式、增长路径或商业化策略。

### 这个场景要解决什么

S8 不是竞品功能截图汇总，而是判断产品应该服务谁、解决什么高价值问题、凭什么赢、怎么增长、怎么赚钱。

### 你应该怎么用

```text
我们在考虑进入企业知识库 AI 助手方向，请帮我从目标市场、用户细分、竞品、差异化定位、商业模式和关键风险做战略分析。
```

### 工作流会怎么走

1. 明确市场边界和目标客户。
2. 识别细分市场和滩头市场。
3. 分析竞品、替代方案和进入壁垒。
4. 定义价值主张和定位。
5. 梳理商业模式和定价假设。
6. 识别战略风险和验证路径。

### 什么时候不能继续

- 市场定义过大，无法落地。
- 只比较功能，不比较用户、渠道、商业模式和壁垒。
- 没有明确目标客户。
- 没有说明为什么我们有机会赢。

## S9：上线准备和 GTM

### 你可能正遇到的情况

功能开发接近完成，准备发布、灰度、对外宣传、销售培训、客服支持或正式上线。

### 这个场景要解决什么

S9 的核心是确认“研发完成不等于上线完成”。上线需要产品、研发、测试、运营、销售、客服、数据和风险预案一起准备好。

### 你应该怎么用

```text
这个功能准备下周上线，请帮我按上线准备和 GTM 场景检查：发布范围、灰度策略、监控指标、回滚方案、销售客服话术和上线后复盘计划。
```

### 工作流会怎么走

1. 明确上线目标和发布范围。
2. 检查功能、数据、权限、文案、帮助文档和运营物料。
3. 设计灰度、监控和回滚。
4. 准备 GTM：目标用户、价值主张、渠道、销售客服材料。
5. 明确上线后观察期和复盘方式。

### 什么时候不能继续

- 没有灰度策略。
- 没有监控指标。
- 没有回滚方案。
- 销售和客服不知道怎么解释。
- 上线后没人负责观察和复盘。

## S10：PM 日常运营协作

### 你可能正遇到的情况

你要整理会议纪要、同步进展、做 sprint plan、复盘、OKR、行动项、风险清单或跨团队协作记录。

### 这个场景要解决什么

S10 的重点不是写漂亮纪要，而是把日常协作变成可追踪的行动、决策和风险。会议结束后，谁负责什么、什么时候完成、哪些问题未决，必须清楚。

### 你应该怎么用

```text
请把这次需求评审会议整理成 PM 日常运营产物：结论、关键决策、行动项、风险、开放问题和下次跟进事项。
```

### 工作流会怎么走

1. 提取会议目标和上下文。
2. 区分讨论内容、结论、决策、行动项和开放问题。
3. 标记负责人和截止时间。
4. 更新风险清单和决策记录。
5. 如果涉及需求变更，提示是否需要回到对应产品场景。

### 什么时候不能继续

- 会议纪要只有流水账。
- 行动项没有负责人。
- 决策没有理由。
- 风险没有后续处理人。
- 需求变更没有同步到 PRD 或路线图。

## 与 UI 和研发智能体的衔接

未来我们的产品工作流会和 UI 智能体、研发智能体串起来。PM Superpowers 主要负责产品设计和需求梳理，UI 智能体负责打磨界面和交互，研发智能体负责技术方案和具体开发。

交接给 UI 前，产品产物至少要说清：

- 目标用户和使用场景。
- 核心任务和用户路径。
- 页面、状态、异常、空状态和权限差异。
- 关键文案和信息优先级。
- 哪些内容可以由 UI 探索，哪些产品规则不能改。

交接给研发前，产品产物至少要说清：

- 需求范围和非目标。
- 功能规则、数据规则、权限规则和异常规则。
- 验收标准和测试场景。
- 埋点、指标和监控口径。
- 依赖、风险、上线方式和回滚方案。

如果这些内容不清楚，插件应该提示 `BLOCKED` 或 `PASS_WITH_RISKS`，而不是强行交接。

## 常见问题

### 安装后为什么当前 thread 里看不到插件？

安装或更新插件后，需要新开一个 Codex thread。旧 thread 通常不会自动加载新插件技能。

### 还需要单独安装 pm-skills 吗？

不需要。PM Superpowers 已经内置 68 个 PM 方法技能，普通用户只需要安装 PM Superpowers 一个插件。

### 为什么插件目录里有很多 skills？

这是正常的。PM Superpowers 由场景/治理技能和底层 PM 方法技能组成。Codex 需要这些技能以独立目录存在，才能按场景自动触发和组合。

### 为什么有时候插件不直接写 PRD？

因为不是所有场景都应该先写 PRD。很多时候应该先做澄清、探索、用户研究、反馈聚类、优先级判断或指标定义。过早写 PRD 会把没想清楚的问题传给下游。

### PRD 模板是不是固定的？

不是。PM Superpowers 的 PRD 应该按需求类型自适应。轻量需求、后台能力、UI 体验、数据接口、优化实验、上线运营所需模块不同。不适用的模块应该标注原因，而不是硬填。

### 更新后如何知道新版本生效？

运行：

```bash
codex plugin list
```

查看 `pm-superpowers@pm-superpowers-internal` 的版本号。更新后请新开 Codex thread，让新版本技能重新加载。

## 推荐学习路径

第一步，先完成安装，并在新 thread 里让插件判断一个真实需求属于哪个场景。

第二步，学习 S0 到 S4：模糊需求澄清、新产品探索、已有产品优化、反馈转路线图、PRD 标准化。这五个场景覆盖大部分日常需求工作。

第三步，学习 S5 到 S10：用户研究、优先级路线图、指标实验、战略商业、上线 GTM、日常运营。这些场景会让产品工作从单点需求变成完整闭环。

第四步，在真实项目中使用产品项目工作区，把 `AGENTS.md`、产品上下文、决策记录、证据表、风险清单和交接检查沉淀下来。

## 最后提醒

使用 PM Superpowers 时，请尽量提供真实上下文，而不是只给一句“帮我写 PRD”。插件越清楚你的业务背景、目标用户、当前证据、想支持的决策和下游对象，输出就越接近真正可用的产品产物。

如果插件提示 `BLOCKED`，先补信息；如果提示 `PASS_WITH_RISKS`，先看风险能否接受；如果提示可以进入下游，再把产物交给 UI 或研发继续推进。
