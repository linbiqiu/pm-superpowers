这份文档给内部产品经理、刚接触产品工作的同事、业务转产品/项目转产品的同事使用。每个要使用 PM Superpowers 的同事，都需要先在自己的电脑或 Codex 环境里完成一次安装，并在团队通知升级时按步骤更新。

安装和更新不需要理解技术原理，照着下面步骤复制执行即可。装好以后，你只需要把真实业务背景说清楚，PM Superpowers 会帮你判断当前是什么产品场景，再带你走对应的产品工作流。

## 先安装并确认可用

第一次使用前，每个人都需要安装一次。下面步骤不需要理解命令含义，复制到终端执行即可。

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>如果你不熟悉终端：</b>可以直接把本节截图或复制给团队工具负责人，让对方协助你完成安装。安装好以后，日常使用不需要再碰这些命令。</p>
</callout>

### 第一步：确认 Codex 可用

打开终端，执行：

```bash
codex plugin --help
```

如果能看到帮助信息，说明可以继续安装。
如果提示找不到 `codex` 命令，请先联系团队工具负责人协助处理。

### 第二步：安装 PM Superpowers

继续在终端执行下面两行：

```bash
codex plugin marketplace add linbiqiu/pm-superpowers --ref main
codex plugin add pm-superpowers@pm-superpowers-internal
```

### 第三步：检查是否安装成功

执行：

```bash
codex plugin list
```

如果看到类似下面的信息，就说明安装成功：

```text
pm-superpowers@pm-superpowers-internal  installed, enabled
```

### 第四步：新开一个对话窗口

安装完成后，请新开一个 Codex 对话窗口。已经打开的旧窗口通常不会自动加载刚安装的能力。

## 如何更新

当团队通知 PM Superpowers 有新版本时，每个使用者都需要在自己的环境里更新一次。

### 推荐更新方式

在终端执行：

```bash
codex plugin marketplace upgrade pm-superpowers-internal
codex plugin add pm-superpowers@pm-superpowers-internal
```

更新完成后，请新开一个 Codex 对话窗口，让新版本生效。

### 如果你本地已经下载过插件仓库

也可以在仓库根目录执行：

```bash
scripts/check_pm_superpowers_update.sh
scripts/update_pm_superpowers.sh
```

执行完成后，同样需要新开一个 Codex 对话窗口。

## 安装更新遇到问题怎么办

| 遇到的问题 | 处理方式 |
| --- | --- |
| 不知道终端在哪里 | 找团队工具负责人协助安装 |
| `codex` 命令找不到 | 说明 Codex CLI 没准备好，找工具负责人处理 |

| 更新后行为还是旧版本 | 新开 Codex 对话窗口；旧窗口通常不会自动加载新版本 |

<callout emoji="❗" background-color="light-yellow" border-color="yellow">

## 它到底帮你做什么

PM Superpowers 可以理解成一个产品经理工作流助手。它不是一个“帮你写漂亮文档”的工具，而是帮你把产品工作按正确顺序推进。

它主要做四件事：

- 判断场景：你现在是在做需求澄清、新产品探索、已有功能优化、PRD、用户研究、上线准备，还是日常协作。
- 追问缺口：如果用户、问题、证据、目标、范围、指标没想清楚，它会先提醒你补齐。
- 规范产物：输出中文产品产物，例如产品 brief、PRD、路线图、指标方案、上线清单、会议纪要。
- 保护下游：在信息不足时不强行进入 UI 或研发，避免把模糊需求传给设计、研发、测试和运营。

<callout emoji="✅" background-color="light-green" border-color="green">
  <p><b>最重要的变化：</b>以前很多人接到需求后第一反应是“写 PRD”。现在应该先判断场景，再决定下一步是不是 PRD。</p>
</callout>

## 它不是什么

为了避免误用，先把边界说清楚。

| 它不是 | 正确理解 |
| --- | --- |
| 不是 PRD 自动生成器 | 它会先判断是否真的适合写 PRD |
| 不是提示词合集 | 它是一套按场景推进的产品工作流 |
| 不是替产品经理做所有判断 | 它会帮助你澄清、追问、结构化和检查 |
| 不是技术工具说明书 | 普通用户只需要会描述业务，不需要懂安装命令 |
| 不是为了让文档更长 | 它的目标是让产物更清楚、更可评审、更可交接 |

## 适合谁使用

- 刚接触产品工作的同事：用它学习产品工作应该先判断场景，而不是直接写文档。
- 日常负责需求和迭代的产品经理：用它提升需求质量、评审质量和交接质量。
- 需要处理大量反馈的产品经理：用它把客户反馈、销售需求、客服工单整理成需求池和路线图候选。
- 需要和设计、研发协作的产品经理：用它检查产物是否已经具备交接条件。
- 产品负责人：用它统一团队工作方式，让新人和老同事都按同一套场景流程推进。

## 第一次怎么开始

如果团队工具负责人已经帮你开通好 PM Superpowers，你的使用方式很简单。

1. 打开产品智能体或 Codex。
2. 描述你的真实业务背景。
3. 让它先判断场景，不要一上来要求写完整 PRD。
4. 如果它提示缺少信息，先补信息。
5. 等门禁通过后，再进入 PRD、UI、研发或上线准备。

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>你不需要说技能名。</b>不要纠结应该调用哪个技能，也不要记工作流名称。你只要描述“我现在遇到什么产品工作”，工具会自动判断。</p>
</callout>

## 第一次可以这样描述

下面这些不是固定提示词，只是帮助你理解应该怎么把业务说清楚。

| 你遇到的情况 | 可以这样描述 |
| --- | --- |
| 只有一个模糊想法 | 我现在有一个比较粗的产品想法，还不知道该怎么推进，请先帮我判断属于哪个产品场景，以及继续前缺什么信息。 |
| 想做新功能 | 我们想做一个面向销售团队的 AI 客户跟进助手，目前只是想法，请先帮我做产品探索，不要直接写 PRD。 |
| 现有功能效果不好 | 这个功能上线后使用率很低，用户也反馈不好用，请先帮我诊断原因和优化方向。 |
| 有很多反馈 | 这里有一批客户反馈，请帮我按用户问题聚类，区分事实、观点和假设，再整理成路线图候选。 |
| 已经准备写需求 | 请帮我把这个需求整理成可评审 PRD，并检查是否可以交给 UI 和研发。 |
| 准备上线 | 这个功能下周准备上线，请帮我检查发布范围、灰度、监控、回滚、销售客服准备和上线后复盘计划。 |

## 不推荐的用法

这些用法会让输出变得表面完整，但实际不可靠。

| 不推荐 | 为什么 |
| --- | --- |
| 直接说“帮我写个 PRD” | 可能还没确认用户、问题、证据和范围 |
| 直接说“帮我设计功能” | 可能只是方案冲动，还没验证问题是否成立 |
| 直接把客户要求当需求 | 客户说的功能不一定是真实问题 |
| 只说“提升效率” | 太泛，无法判断场景、指标和成功标准 |
| 跳过工具提示的缺口 | 会把没想清楚的问题传给下游 |

## 先回答这五个问题

每次开始前，先尽量用自己的话回答这五个问题。答不完整也没关系，PM Superpowers 会继续追问。

| 问题 | 通俗解释 | 答不清会有什么风险 |
| --- | --- | --- |
| 这件事从哪里来？ | 老板想法、客户反馈、指标异常、会议结论、竞品压力，还是上线任务？ | 无法判断应该走哪个场景 |
| 这件事服务谁？ | 谁是用户、客户、内部角色或下游团队？ | 容易做出没人真正需要的需求 |
| 你现在要做什么决定？ | 决定做不做、先做哪个、怎么做、能不能上线，还是怎么复盘？ | 输出会变成泛泛分析 |
| 当前最不确定的是什么？ | 用户、问题、证据、方案、范围、指标、风险，哪个最不清楚？ | 容易跳过关键思考 |
| 产物要给谁用？ | 给自己继续探索、给设计、给研发、给销售客服，还是给管理层？ | 输出格式和深度会不匹配 |

<callout emoji="📌" background-color="light-yellow" border-color="yellow">
  <p><b>判断规则：</b>如果这五个问题里有两个以上答不清，不要直接进入 PRD。先做模糊需求澄清。</p>
</callout>

## 为什么有时会提示 BLOCKED

`BLOCKED` 可以理解成“现在还不能继续往下走”。这不是工具坏了，也不是你用错了，而是在提醒你：继续推进会有较大风险。

以下情况就应该停下来：

- 不知道用户是谁。
- 不知道问题是什么。
- 不知道要支持什么决策。
- 没有证据，却要求给确定结论。
- 没有范围和验收标准，却要交给研发。
- 没有关键状态和用户流，却要交给 UI。
- 没有监控和回滚方案，却要上线。

<callout emoji="❌" background-color="light-red" border-color="red">
  <p><b>最危险的情况：</b>信息没想清楚，但文档写得很完整。这样的产物看起来专业，实际会把错误传给设计、研发、测试、销售和客服。</p>
</callout>

## S0：模糊需求澄清

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>当输入还只是一句话想法或混合诉求时，先不要写 PRD，而是把它拆成清楚的产品任务。</p>
</callout>

### 你可能正遇到的情况

- 老板说：“我们也要做一个 AI 助手。”
- 客户说：“我们想要一个报表。”
- 销售说：“这个功能客户很急。”
- 业务说：“竞品有这个，我们是不是也要做？”
- 团队说：“帮我看看这个产品怎么做。”

这些话都不是清楚需求，只是产品工作的入口。

### 这个场景真正要解决什么

S0 的目标不是马上给答案，而是判断“下一步该做什么”。也就是说，它会帮你把模糊输入变成一个清楚的产品任务。

你最终应该知道：

- 这件事的来源是什么。
- 谁是可能的用户或客户。
- 当前已知事实有哪些。
- 哪些只是猜测。
- 应该进入哪个后续场景。
- 继续推进前必须补齐什么。

### 怎么判断产物合格

一个合格的 S0 产物应该能回答：

- 这不是哪个场景，为什么。
- 更像哪个场景，为什么。
- 当前有什么证据。
- 当前最大缺口是什么。
- 下一步应该问谁、看什么数据、做什么调研或进入哪个工作流。

## S1：新产品或新功能探索

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>新想法先证明“问题值得解决”，再讨论“功能怎么做”。</p>
</callout>

### 你可能正遇到的情况

团队想做一个新产品、新模块、新 AI 能力、新商业化能力或新用户场景。大家觉得方向有价值，但还不知道用户是不是真的需要。

### 这个场景真正要解决什么

S1 不是把想法写成需求，而是证明这个想法是否值得继续。重点是把热情变成假设，再把假设变成验证计划。

你要想清楚：

- 谁会在什么情况下需要它。
- 用户现在怎么解决这个问题。
- 现有解决方式哪里不好。
- 用户为什么会愿意切换到我们的方案。
- 最危险的假设是什么。
- 最低成本的验证方式是什么。

### 什么时候不能进入 PRD

- 目标用户只写“所有人”或“所有销售”。
- 问题只写“提升效率”，没有具体场景。
- 没有替代方案分析。
- 没有关键假设排序。
- MVP 只是功能合集，看不出要验证什么。

## S2：已有产品优化

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>现有功能效果不好时，先诊断原因和指标基线，再决定优化方案。</p>
</callout>

### 你可能正遇到的情况

已有功能上线了，但转化率、留存、使用率、满意度或投诉情况不理想。用户说难用，业务说效果不好，团队想做一轮优化。

### 这个场景真正要解决什么

S2 的核心不是马上改版，而是先诊断原因。问题可能来自用户路径、信息架构、功能价值、数据口径、运营触达，也可能是目标用户本身判断错了。

插件会帮你梳理：

- 当前症状是什么。
- 哪个用户路径出问题。
- 有没有基线指标。
- 用户反馈和数据是否指向同一个问题。
- 哪些原因是证据支持，哪些只是猜测。
- 应该先做哪个优化实验。

### 什么时候不能继续

- 没有当前指标或用户反馈。
- 不知道问题发生在哪个路径。
- 把“用户说难用”直接等同于“必须重做 UI”。
- 没有判断优化是否有效的指标。

## S3：用户反馈转路线图

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>把杂乱反馈还原成用户问题和证据，再变成可排序、可取舍的路线图候选。</p>
</callout>

### 你可能正遇到的情况

你手上有很多客户反馈、销售需求、客服工单、老板要求或竞品参考。大家都说重要，但资源有限，必须决定哪些进入路线图。

### 这个场景真正要解决什么

S3 的重点是把“谁都想要”的请求，变成“可以判断和取舍”的需求池。

插件会帮你区分：

- 用户原话是什么。
- 真实痛点是什么。
- 客户提出的是问题，还是解决方案。
- 这个反馈出现了多少次。
- 影响哪些用户或客户。
- 是否支撑当前业务目标。
- 进入路线图、暂缓或拒绝的理由是什么。

### 什么时候不能继续

- 反馈没有来源。
- 只记录“客户要什么”，没有记录“为什么要”。
- 只按客户级别或声音大小排序。
- 没有说明哪些需求不做以及原因。

## S4：PRD 标准化

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>PRD 的目标不是内容多，而是让设计、研发和测试不用猜产品决策。</p>
</callout>

### 你可能正遇到的情况

你已经准备把需求交给设计、研发或测试，或者已有 PRD 结构混乱，需要重写、补齐和评审。

### 这个场景真正要解决什么

S4 的目标是把需求写到“可以评审、可以执行、可以验收”的程度。不是所有 PRD 都需要同一套大模板，轻量需求、后台能力、数据接口、体验优化、上线运营所需内容都不一样。

插件会帮你检查：

- 用户和问题是否清楚。
- 范围和非目标是否清楚。
- 规则、边界和异常状态是否清楚。
- 验收标准是否可执行。
- 是否需要埋点、权限、数据口径或运营说明。
- 是否已经适合交给 UI 或研发。

### 什么时候不能交给下游

- 只写了解决方案，没有写用户问题。
- 没有范围和非目标。
- 没有验收标准。
- 没有边界状态、异常状态和权限规则。
- 交给 UI 但没有关键状态和用户流。
- 交给研发但没有数据、接口、规则或验收口径。

## S5：用户研究

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>用户研究不是问用户想要什么功能，而是理解真实行为、动机、阻碍和替代方案。</p>
</callout>

### 你可能正遇到的情况

你要访谈用户、验证需求、理解流失原因、总结访谈记录，或者把定性研究转成产品决策。

### 这个场景真正要解决什么

S5 帮你把用户研究从“聊天”变成“支持产品决策的证据收集”。

它会帮你明确：

- 这次研究要支持什么决策。
- 应该访谈哪类用户。
- 访谈问题是否带有诱导。
- 如何记录用户原话。
- 如何从原话中提炼洞察。
- 洞察应该进入需求、假设、路线图，还是继续验证。

### 什么时候不能继续

- 访谈问题引导用户回答。
- 只问“你想不想要这个功能”。
- 没有招募标准。
- 访谈总结只有观点，没有用户原话和证据。
- 洞察没有连接到产品决策。

## S6：优先级和路线图

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>路线图不是功能排队，而是围绕产品结果做透明取舍。</p>
</callout>

### 你可能正遇到的情况

需求很多，资源有限。业务、客户、研发、老板都有诉求，你需要决定先做什么、后做什么、为什么不做某些事。

### 这个场景真正要解决什么

S6 不是为了找一个看起来客观的打分表，而是让团队清楚每个取舍背后的理由。

插件会帮你整理：

- 当前业务目标是什么。
- 每个需求服务哪个目标。
- 用户价值和业务价值分别是什么。
- 成本、风险、依赖是什么。
- 哪些需求应该先做。
- 哪些需求应该暂缓或拒绝。
- 路线图如何围绕结果而不是围绕功能展开。

### 什么时候不能继续

- 没有业务目标。
- 候选需求粒度不一致。
- 只按老板优先级或客户级别排序。
- 没有说明为什么暂缓或拒绝。
- 路线图只有功能，没有结果指标。

## S7：指标和实验

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>先定义用什么判断好坏，再谈上线、实验和复盘。</p>
</callout>

### 你可能正遇到的情况

你要定义北极星指标、功能指标、埋点方案、实验方案，或者判断一次改动是否真的有效。

### 这个场景真正要解决什么

S7 帮团队回答一个关键问题：我们到底用什么判断这件事做得好不好。

插件会帮你明确：

- 核心目标是什么。
- 哪些指标代表结果。
- 哪些指标代表过程。
- 哪些指标是护栏，防止局部优化伤害整体体验。
- 当前有没有基线。
- 实验是否可行。
- 成功、失败、继续观察分别怎么判断。

### 什么时候不能继续

- 指标只写“提升体验”。
- 没有基线。
- 没有护栏指标。
- 实验样本不足却要求确定结论。
- 埋点事件没有定义触发时机和属性。

## S8：战略和商业模式

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>战略分析不是竞品功能截图汇总，而是判断服务谁、凭什么赢、怎么增长和怎么赚钱。</p>
</callout>

### 你可能正遇到的情况

你需要判断一个方向是否值得做，分析市场、竞品、定位、商业模式、增长路径或商业化策略。

### 这个场景真正要解决什么

S8 帮你从更高层面判断一个方向是否值得投入，而不是只比较竞品有什么功能。

插件会帮你思考：

- 目标市场到底是什么。
- 哪个用户细分最值得先做。
- 竞品和替代方案是什么。
- 我们的差异化价值在哪里。
- 用户为什么会选择我们。
- 商业模式和定价假设是什么。
- 最大战略风险是什么。

### 什么时候不能继续

- 市场定义过大，无法落地。
- 只比较功能，不比较用户、渠道、商业模式和壁垒。
- 没有明确目标客户。
- 没有说明为什么我们有机会赢。

## S9：上线准备和 GTM

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>研发完成不等于上线完成，上线还需要灰度、监控、回滚、运营和销售客服准备。</p>
</callout>

### 你可能正遇到的情况

功能开发接近完成，准备发布、灰度、对外宣传、销售培训、客服支持或正式上线。

### 这个场景真正要解决什么

S9 帮你确认上线前后的准备是否完整。上线不是把功能放出去，而是让用户、内部团队和系统风险都被照顾到。

插件会帮你检查：

- 发布范围是什么。
- 是否需要灰度。
- 上线后看哪些指标。
- 出问题如何回滚。
- 销售和客服是否知道怎么解释。
- 用户是否需要帮助文档或公告。
- 上线后什么时候复盘。

### 什么时候不能继续

- 没有灰度策略。
- 没有监控指标。
- 没有回滚方案。
- 销售和客服不知道怎么解释。
- 上线后没人负责观察和复盘。

## S10：PM 日常运营协作

<callout emoji="💡" background-color="light-blue" border-color="blue">
  <p><b>一句话理解：</b>会议和协作产物的价值不是记录流水账，而是沉淀决策、行动项、风险和开放问题。</p>
</callout>

### 你可能正遇到的情况

你要整理会议纪要、同步进展、做 sprint plan、复盘、OKR、行动项、风险清单或跨团队协作记录。

### 这个场景真正要解决什么

S10 帮你把日常协作变成可追踪的管理资产。会议结束后，谁负责什么、什么时候完成、哪些问题未决，必须清楚。

插件会帮你整理：

- 会议目标和背景。
- 已达成的结论。
- 关键决策和决策理由。
- 行动项、负责人、截止时间。
- 风险和开放问题。
- 是否需要回到某个产品场景继续处理。

### 什么时候不能继续

- 会议纪要只有流水账。
- 行动项没有负责人。
- 决策没有理由。
- 风险没有后续处理人。
- 需求变更没有同步到 PRD 或路线图。

## 与 UI 和研发的衔接

PM Superpowers 负责把产品问题和需求讲清楚。后续可以交给 UI 智能体或设计同事打磨界面，也可以交给研发智能体或研发同事做技术方案和开发。

交接前至少要说清这些内容：

| 交接对象 | 交接前必须说清 | 如果缺失会怎样 |
| --- | --- | --- |
| UI / 设计 | 目标用户、使用场景、核心任务、用户路径、页面状态、异常状态、空状态、权限差异、关键文案、信息优先级 | 设计只能做视觉包装，无法判断真正的交互重点 |
| 研发 | 需求范围、非目标、功能规则、数据规则、权限规则、异常规则、验收标准、测试场景、埋点口径、监控口径、依赖、风险、上线方式、回滚方案 | 研发需要反复追问，或按自己的理解补产品决策 |

<callout emoji="✅" background-color="light-green" border-color="green">
  <p><b>交接前检查：</b>如果你不能清楚回答“谁在什么场景下完成什么任务”“哪些规则不能变”“怎么验收算完成”，就还不适合交给 UI 或研发。</p>
</callout>

## 常见问题

### 我是不是需要懂技术才能用？

不需要。普通产品经理只需要会描述业务背景、目标用户、问题、证据和想要支持的决策。安装、更新、版本检查由工具负责人处理。

### 我需要记住这些场景编号吗？

不需要。场景编号是为了团队培训和沟通方便。日常使用时，你直接说真实工作内容即可。

### 为什么它有时候不直接写 PRD？

因为不是所有工作都应该先写 PRD。很多时候应该先做澄清、探索、用户研究、反馈聚类、优先级判断或指标定义。过早写 PRD 会把没想清楚的问题传给下游。

### PRD 模板是不是固定的？

不是。PRD 应该按需求类型自适应。轻量需求、后台能力、UI 体验、数据接口、优化实验、上线运营所需模块不同。不适用的模块应该说明原因，而不是硬填。

### 如果工具提示 BLOCKED，我应该怎么办？

先看它列出的缺口。通常需要补目标用户、问题证据、业务目标、范围、验收标准、指标或风险信息。补齐后再继续。

### 如果我不知道该怎么描述怎么办？

可以先说：“我现在不知道这件事该怎么推进，请先帮我判断场景，并告诉我需要补哪些信息。”这就是最适合新手的起点。

## 推荐学习路径

<checkbox done="false">第一步：先拿一个真实需求，让 PM Superpowers 判断它属于哪个场景。</checkbox>
<checkbox done="false">第二步：重点学习 S0-S4。模糊需求澄清、新产品探索、已有产品优化、反馈转路线图、PRD 标准化覆盖大部分日常需求工作。</checkbox>
<checkbox done="false">第三步：继续学习 S5-S10。用户研究、优先级路线图、指标实验、战略商业、上线准备、日常运营会让产品工作从单点需求变成完整闭环。</checkbox>
<checkbox done="false">第四步：在真实项目中沉淀产品上下文、关键决策、证据、风险和交接检查。</checkbox>

## 最后提醒

使用 PM Superpowers 时，请尽量提供真实上下文，而不是只给一句“帮我写 PRD”。它越清楚你的业务背景、目标用户、当前证据、想支持的决策和下游对象，输出就越接近真正可用的产品产物。

<callout emoji="📌" background-color="light-gray" border-color="gray">
  <p><b>一句话收尾：</b>先判断场景，再选择工作流；先补齐证据，再写产品产物；先通过门禁，再交给下游。</p>
</callout>
