# PM Superpowers 使用手册

这份文档给产品经理和刚接触产品工作的同事使用。它不要求你先懂所有产品方法论，而是按常见工作场景告诉你：现在处在什么场景、应该走什么流程、要准备什么输入、最后应该得到什么结果。

## 先理解这个插件在做什么

PM Superpowers 不是单个提示词，也不是替你直接写一份看起来完整的 PRD。它是一个产品工作流插件，作用是：

- 自动判断你当前属于哪类产品工作场景。
- 帮你选择合适的产品工作流和底层 PM skills，这些底层方法技能已经随插件内置。
- 在信息没想清楚时阻止你直接进入下一步。
- 按公司统一规范输出中文产品简报、PRD、路线图、指标方案、上线清单等。
- 把产品产物整理到可以交给 UI 智能体和研发智能体继续工作的程度。

使用时，你可以直接说自然语言，例如：

```text
我们想做一个面向销售团队的 AI 周报功能，帮我梳理一下产品思路。
```

插件会先判断场景，而不是直接写 PRD。如果缺少目标用户、问题、证据或决策，它会先要求补齐。

## 推荐使用方式

1. 先描述业务背景、目标用户和你想解决的问题。
2. 如果不知道该走什么流程，直接让插件判断工作流。
3. 不要一开始就要求写最终文档，先让插件做场景识别和门禁检查。
4. 当插件提示 `BLOCKED` 时，先补齐缺失信息，不要绕过。
5. 当产物要交给设计或研发时，运行下游交接检查。

## 如何深入学习使用场景

这份使用手册是快速入口。如果要做团队培训、规范落地或新人学习，请继续阅读：

- [PM_SUPERPOWERS_SCENARIO_DEEP_DIVE.md](PM_SUPERPOWERS_SCENARIO_DEEP_DIVE.md)

深度手册会逐个讲清楚每个场景的业务背景、产品经理真正要解决的问题、必须输入、完整工作流、阻断条件、好产物标准、下游交接和示例输入。

插件面向中国团队使用。除非你明确要求英文或双语，插件默认会以中文输出结论、表格字段、模板标题和交付产物。

如果你希望把一次需求讨论沉淀成可持续管理的产品项目，可以让插件初始化产品项目工作区：

```text
请用 PM Superpowers 初始化当前项目的产品工作区，创建 AGENTS.md、产品上下文、决策记录、证据表、风险清单和下游交接检查。
```

## 场景 0：需求很模糊，不知道该怎么开始

适合使用在：

- 你只有一个想法，但还没想清楚用户和问题。
- 领导或业务同事提出一句话需求。
- 你不知道应该写 PRD、做调研、看数据还是排路线图。
- 需求里混着用户反馈、老板想法、商业目标和技术方案。

典型说法：

```text
帮我看看这个产品需求应该怎么推进。
```

插件会做：

- 澄清当前任务到底要支持什么决策。
- 识别目标用户、产品阶段、已有证据和缺口。
- 判断应该进入十个场景中的哪一个。
- 如果信息不足，先给出最少需要补齐的问题。

常见输出：

- 澄清后的产品 brief。
- 推荐场景和下一步技能。
- 证据和假设清单。
- 门禁状态：`PASS`、`PASS_WITH_RISKS` 或 `BLOCKED`。

关键原则：如果没有用户、问题和决策目标，不应该直接写 PRD。

## 场景 1：新产品或新功能探索

适合使用在：

- 想做一个新产品、新模块或新功能。
- 有一个机会点，但还不知道是否值得做。
- 想从 0 到 1 梳理用户、问题、价值和验证方式。
- 准备进入 PRD 或设计前，需要先判断问题是否成立。

典型说法：

```text
我们想做一个给客服主管看的质检看板，帮我做产品探索。
```

核心流程：

1. 明确目标用户和使用场景。
2. 说清楚用户问题和业务目标。
3. 把已有证据和假设分开。
4. 画出机会、解决方案和实验。
5. 找出最高风险假设。
6. 设计验证计划。
7. 通过问题门禁后再进入 PRD 或 UI。

涉及技能：

- `new-product-discovery`
- `evidence-classifier`
- `pm-gate-review`
- 底层 PM skills：`user-segmentation`、`job-stories`、`opportunity-solution-tree`、`identify-assumptions-new`、`prioritize-assumptions`、`brainstorm-experiments-new`、`value-proposition`

标准输出：

- 产品探索 brief。
- 机会解决方案树。
- 假设地图。
- 验证实验计划。

容易犯的错：

- 用户是谁不清楚就开始设计功能。
- 把领导想法当成用户证据。
- 没有验证计划就直接写 PRD。

## 场景 2：已有产品优化

适合使用在：

- 现有功能转化率、留存、使用率或满意度不好。
- 用户反馈某个流程难用。
- 你需要判断是体验问题、产品策略问题还是数据问题。
- 想做一轮优化实验。

典型说法：

```text
导出功能使用率很低，用户也经常投诉，帮我分析怎么优化。
```

核心流程：

1. 明确要优化的功能和用户路径。
2. 确定当前症状和基线指标。
3. 收集用户反馈、数据和业务约束。
4. 诊断可能原因。
5. 生成优化方案和实验。
6. 评估收益、成本和风险。
7. 通过指标门禁后进入 PRD 或迭代计划。

涉及技能：

- `existing-product-optimization`
- `metrics-experiments`
- `evidence-classifier`
- 底层 PM skills：`customer-journey-map`、`metrics-dashboard`、`cohort-analysis`、`brainstorm-ideas-existing`、`brainstorm-experiments-existing`、`pre-mortem`

标准输出：

- 现状诊断。
- 指标计划。
- 优化实验计划。
- 优化需求候选列表。

关键原则：没有基线指标时，可以先补埋点或做定性诊断，但不要声称优化会带来确定收益。

## 场景 3：用户反馈转路线图

适合使用在：

- 有一堆客户反馈、销售需求、客服工单或老板要求。
- 需要把杂乱请求整理成主题和机会。
- 需要决定哪些进入路线图，哪些暂缓。
- 想避免谁声音大谁优先。

典型说法：

```text
这里有 30 条客户反馈，帮我整理成需求池并排优先级。
```

核心流程：

1. 规范化反馈来源、用户类型、痛点和请求。
2. 区分用户原话、信号、假设和意见。
3. 把相似反馈聚类成主题。
4. 把解决方案请求还原成用户问题。
5. 按影响、频率、业务价值和风险排序。
6. 形成路线图候选项。
7. 记录哪些需求被选择、延期或拒绝。

涉及技能：

- `feedback-to-roadmap`
- `evidence-classifier`
- `prioritization-roadmap`
- 底层 PM skills：`analyze-feature-requests`、`sentiment-analysis`、`prioritize-features`、`outcome-roadmap`、`stakeholder-map`

标准输出：

- 反馈主题表。
- 证据表。
- 路线图候选项。
- 取舍说明和决策记录。

关键原则：客户提出的功能方案不等于真正的问题，必须先还原问题。

## 场景 4：PRD 标准化

适合使用在：

- 要从零写 PRD。
- 已有 PRD 但结构混乱，需要重写。
- 想检查 PRD 是否可以交给设计或研发。
- 需要补用户故事、验收标准、边界条件和埋点。

典型说法：

```text
帮我把这个需求整理成可以评审的 PRD。
```

核心流程：

1. 确认 PRD 的读者和用途。
2. 检查问题、用户、证据、目标、范围和指标。
3. 标准化 PRD 结构。
4. 写清楚用户故事或 job stories。
5. 补齐验收标准、边界状态和非目标。
6. 做门禁检查。
7. 如果要交给 UI 或研发，做下游 readiness 检查。

涉及技能：

- `prd-standardization`
- `pm-gate-review`
- `downstream-readiness`
- 底层 PM skills：`create-prd`、`job-stories`、`user-stories`、`test-scenarios`、`pre-mortem`

标准输出：

- 标准 PRD。
- 验收标准。
- 风险和开放问题。
- UI/研发交接检查表。

关键原则：PRD 的目标不是写得完整，而是让后续团队不用猜产品决策。

## 场景 5：用户研究

适合使用在：

- 准备访谈用户。
- 想验证一个需求或假设。
- 已经有访谈记录，需要总结洞察。
- 需要把定性研究转成产品决策。

典型说法：

```text
我想访谈 8 个新用户，验证他们为什么没有完成首次配置。
```

核心流程：

1. 明确研究要支持的决策。
2. 确定目标受访者。
3. 写研究问题和访谈脚本。
4. 访谈后提取用户原话、信号和假设。
5. 总结洞察和产品含义。
6. 判断下一步进入探索、PRD、路线图还是实验。

涉及技能：

- `user-research`
- `evidence-classifier`
- 底层 PM skills：`interview-script`、`summarize-interview`、`user-personas`、`customer-journey-map`、`identify-assumptions-new`

标准输出：

- 研究计划。
- 访谈脚本。
- 访谈洞察总结。
- 假设验证清单。

关键原则：没有决策目的的访谈很容易变成泛泛聊天。

## 场景 6：优先级和路线图

适合使用在：

- 候选需求很多，需要排序。
- 要制定季度路线图或版本计划。
- 各部门对优先级有冲突。
- 想把路线图从功能列表升级成目标导向。

典型说法：

```text
这些需求都很重要，帮我按季度目标排一个路线图。
```

核心流程：

1. 明确决策周期和资源约束。
2. 把候选项统一成可比较的颗粒度。
3. 选择优先级框架。
4. 显式标注评分依据和假设。
5. 输出 Now / Next / Later 或 outcome roadmap。
6. 记录被延期和放弃的事项。
7. 做路线图门禁。

涉及技能：

- `prioritization-roadmap`
- `pm-decision-log`
- 底层 PM skills：`prioritization-frameworks`、`prioritize-features`、`outcome-roadmap`、`brainstorm-okrs`、`stakeholder-map`

标准输出：

- 优先级评分表。
- outcome roadmap。
- 取舍说明。
- 决策记录。

关键原则：路线图不是承诺所有功能，而是承诺优先解决哪些结果问题。

## 场景 7：指标、数据分析和实验

适合使用在：

- 不知道应该看什么指标。
- 要设计实验或 A/B test。
- 想写 SQL 需求或看 cohort。
- 要定义上线后的观测指标。

典型说法：

```text
这个功能上线后怎么判断有没有效果？帮我设计指标和实验。
```

核心流程：

1. 明确产品问题和决策。
2. 定义主指标、辅助指标和护栏指标。
3. 检查数据源和埋点是否存在。
4. 选择分析、看板、cohort 或实验路径。
5. 定义成功阈值和失败阈值。
6. 输出指标方案或实验方案。
7. 上线后做结果解读。

涉及技能：

- `metrics-experiments`
- `post-launch-learning`
- 底层 PM skills：`north-star-metric`、`metrics-dashboard`、`sql-queries`、`ab-test-analysis`、`cohort-analysis`、`dummy-dataset`

标准输出：

- 指标方案。
- 实验计划。
- SQL 请求或数据需求。
- 实验结果解读。

关键原则：指标必须服务一个产品决策，而不是为了看数据而看数据。

## 场景 8：产品战略和商业模式

适合使用在：

- 要判断产品方向。
- 想分析市场、竞品、定位、商业模式。
- 要确定 ICP、定价或商业化路径。
- 准备做高层评审或战略规划。

典型说法：

```text
我们要不要把产品从中小客户扩展到企业客户，帮我做战略分析。
```

核心流程：

1. 明确战略问题和决策人。
2. 界定市场、客户和产品范围。
3. 分析市场、竞品、客户和商业模式。
4. 给出战略选项和取舍。
5. 找出最关键的不确定性。
6. 输出战略建议。
7. 做 red team 挑战后再进入路线图。

涉及技能：

- `strategy-business-model`
- `pm-decision-log`
- 底层 PM skills：`product-vision`、`product-strategy`、`market-segments`、`market-sizing`、`competitor-analysis`、`swot-analysis`、`business-model`、`pricing-strategy`、`ideal-customer-profile`

标准输出：

- 战略备忘录。
- 市场和竞品分析。
- ICP 和定位。
- 商业模式或定价建议。

关键原则：战略输出必须能指导后续取舍，否则只是分析材料。

## 场景 9：上线准备和 GTM

适合使用在：

- 功能准备上线。
- 要检查上线前是否漏项。
- 要准备 release notes、GTM、销售支持或客服支持。
- 要制定灰度、回滚和监控计划。

典型说法：

```text
这个版本下周上线，帮我检查上线准备和发布计划。
```

核心流程：

1. 确认上线范围、用户和时间窗口。
2. 检查产品、设计、研发、QA、数据、GTM、客服和法务准备。
3. 明确发布节奏、渠道和目标用户。
4. 准备 release notes 和支持材料。
5. 制定风险、回滚和监控计划。
6. 通过上线门禁。
7. 设定发布后复盘时间。

涉及技能：

- `launch-readiness`
- `pm-gate-review`
- `post-launch-learning`
- 底层 PM skills：`gtm-strategy`、`gtm-motions`、`beachhead-segment`、`ideal-customer-profile`、`competitive-battlecard`、`release-notes`、`shipping-artifacts`

标准输出：

- 上线检查表。
- GTM 计划。
- release notes。
- 风险和回滚计划。

关键原则：上线不是研发完成，而是用户、业务、数据和支持体系都准备好。

## 场景 10：PM 日常运营和协作

适合使用在：

- 会议纪要、周会、需求评审、复盘。
- sprint plan、OKR、跨部门对齐。
- 要跟踪决策、行动项和负责人。
- 想把日常协作沉淀为可追踪记录。

典型说法：

```text
帮我把这次需求评审会议整理成结论、行动项和风险。
```

核心流程：

1. 识别会议或项目上下文。
2. 提取事实、决策、承诺、风险和开放问题。
3. 形成会议纪要或 sprint plan。
4. 分配 owner 和时间。
5. 把未解决的产品问题路由到对应场景。

涉及技能：

- `pm-operations`
- `pm-decision-log`
- 底层 PM skills：`summarize-meeting`、`sprint-plan`、`retro`、`stakeholder-map`、`brainstorm-okrs`

标准输出：

- 会议纪要。
- 决策记录。
- 行动项列表。
- 需要进入其他工作流的问题。

关键原则：运营类产物要清楚记录谁决定了什么、谁负责什么、什么时候跟进。

## 什么时候必须先停下来

插件出现 `BLOCKED` 并不是失败，而是在保护团队避免产出虚假的确定性。以下情况应该停下来：

- 不知道目标用户是谁。
- 不知道要支持什么决策。
- 没有问题陈述却要求写 PRD。
- 没有候选项却要求排优先级。
- 没有指标定义却要求分析效果。
- 没有范围和验收标准却要求交给研发。
- 没有用户流和关键状态却要求生成 UI。

## 推荐学习顺序

1. 先学会识别十个场景。
2. 再学会看证据和假设的区别。
3. 再学会使用门禁状态。
4. 最后学习每个场景背后的底层 PM skills。
