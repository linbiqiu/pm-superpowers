# PM 十个高频场景使用手册

本文档用于帮助刚接触产品工作的人，以及公司所有产品经理，理解并掌握十个高频产品工作场景。重点不是记住技能名称，而是学会判断自己处在哪个场景、该采用什么工作流、需要补齐哪些信息、什么时候必须停下来不能进入下一步。

## 如何使用这份手册

每次开始一项产品工作，先问四个问题：

1. 我现在面对的是新想法、已有问题、用户反馈、需求文档、用户研究、优先级、指标、战略、上线，还是日常运营？
2. 我手上的输入是什么：一句想法、数据现象、反馈列表、会议纪要、PRD 草稿、老板目标，还是待上线功能？
3. 我现在缺什么：用户、问题、证据、目标、指标、方案、范围、验收标准，还是风险预案？
4. 如果继续往下一步走，下游是谁：自己继续探索、产品评审、UI 设计、研发、测试、销售、运营，还是管理层？

如果答不清楚，不要直接写 PRD、排期、画 UI 或交给研发。先进入对应场景的澄清和关卡检查。

## 十个场景总览

| 编号 | 场景 | 典型问题 | 应对目标 |
| --- | --- | --- | --- |
| S1 | 新产品/新想法探索 | “我们能不能做一个...” | 判断想法是否值得推进，并形成 MVP PRD |
| S2 | 已有产品问题优化 | “这个功能不好用/指标不好” | 找到真实问题，形成可验证改进方案 |
| S3 | 客户反馈和功能请求整理 | “客户提了很多需求” | 把杂乱反馈变成优先级和路线图 |
| S4 | 写 PRD/需求文档 | “帮我写需求给研发” | 把需求变成可评审、可设计、可开发、可验收的契约 |
| S5 | 用户研究和访谈 | “我要访谈用户/总结访谈” | 设计访谈、提炼洞察、沉淀证据 |
| S6 | 功能优先级和路线图 | “这么多功能先做哪个” | 基于目标和证据做取舍 |
| S7 | 指标和增长实验 | “该看什么指标/实验结果怎么看” | 建立指标体系，设计和判断实验 |
| S8 | 产品战略和商业模式 | “这个方向值不值得做” | 明确方向、市场、价值、商业模式和取舍 |
| S9 | 上线发布和风险检查 | “准备上线，帮我检查” | 确认风险、沟通、测试、发布和监控准备完备 |
| S10 | PM 日常运转 | “会议纪要/OKR/Sprint/复盘” | 让日常协作沉淀为决策、行动项和改进闭环 |

## S1：新产品/新想法探索

### 场景定义

这是从 0 到 1 的探索场景。用户通常只有一个产品想法、业务机会或老板提出的方向，但目标用户、真实问题、市场需求、核心价值和验证方式还不清楚。

### 典型触发信号

- “我有一个新想法。”
- “我们能不能做一个 AI xxx？”
- “这个市场好像有机会。”
- “老板说想做某个方向。”
- “我们想做一个新产品，但不知道从哪里开始。”

### 适合使用

- 新业务探索。
- 创业想法验证。
- 公司内部新产品线。
- 一个还没有明确需求证据的产品概念。
- 需要判断是否值得投入资源。

### 不适合使用

- 已经有明确 PRD，只是需要补充格式。
- 已经验证过需求，只是进入交付。
- 已有产品的小功能优化。

### 关键判断

这个场景最重要的判断不是“能不能做”，而是：

- 目标用户是谁？
- 他们在什么场景下遇到什么问题？
- 这个问题是否足够高频、痛苦、重要？
- 他们现在用什么替代方案解决？
- 哪些假设如果错了，这个产品就不成立？
- 最低成本如何验证？

### 推荐应对流程

1. 澄清想法：一句话说清产品、用户、场景和目标。
2. 发散方案：从 PM、设计、工程视角生成可能方案。
3. 识别假设：列出价值、可用性、可行性、商业、GTM 等风险。
4. 排序假设：按影响和不确定性找出最该验证的假设。
5. 设计实验：为高风险假设设计低成本验证方式。
6. 定义价值主张：说明用户为什么选择你，而不是替代方案。
7. 形成 MVP 范围：只包含验证核心价值所需的最小能力。
8. 输出 MVP PRD：进入后续 UI 或研发前的需求基础。

### 必问问题

- 这个产品服务谁？请具体到角色、行业、规模、使用场景。
- 他们现在怎么解决这个问题？为什么现有方式不够好？
- 这个问题发生频率和痛感有多高？
- 如果只能验证一个假设，最应该验证哪个？
- 你希望 4 到 6 周内证明什么？
- 什么结果会让我们停止这个方向？

### 标准输出

- 产品想法摘要。
- 目标用户和核心场景。
- 关键假设清单。
- 假设优先级。
- 验证实验计划。
- 价值主张。
- MVP 范围。
- MVP PRD。
- 初版成功指标和验收标准。

### 阻断条件

出现以下情况，不能进入 PRD 或研发：

- 目标用户不清楚。
- 用户问题只是主观猜测，没有场景描述。
- 没有替代方案分析。
- 没有关键假设排序。
- 没有验证计划。
- MVP 范围过大，无法说明为什么这些功能是必须的。

### 常见误区

- 一上来就列功能。
- 把“老板想做”当作用户需求。
- 把技术可行当作产品可行。
- 没有验证需求就要求设计 UI。
- MVP 变成“大而全的一期”。

### 示例提示词

```text
我有一个新想法：面向销售团队的 AI 客户跟进助手。请按新产品探索工作流推进。先判断目标用户和核心场景是否清楚，如果不清楚先追问；然后帮我识别关键假设、排序假设、设计验证实验，最后再决定是否进入 MVP PRD。
```

### 新人要掌握

新想法不是从写功能开始，而是从用户、问题、假设和验证开始。任何没有验证路径的新想法，都只能算概念，不能算需求。

## S2：已有产品问题优化

### 场景定义

这是对已有产品、已有功能或已有流程进行诊断和改进的场景。通常已经有用户、功能和数据，但某个体验、转化、留存、效率或满意度不达预期。

### 典型触发信号

- “这个功能用户用不起来。”
- “转化率下降了。”
- “用户说流程太复杂。”
- “我们想优化 onboarding。”
- “这个页面需要改版。”
- “客户觉得不好用。”

### 适合使用

- 已有产品功能改进。
- 关键指标异常。
- 用户体验差。
- 转化、留存、活跃、满意度下降。
- 想把模糊问题转成明确优化方案。

### 不适合使用

- 完全没有现有产品或现有用户。
- 只是视觉层面小改，不涉及用户目标和指标。
- 已经有明确方案，只需要写 UI 文案。

### 关键判断

这个场景最容易被“方案先行”带偏。关键是先区分：

- 现象是什么？
- 真正的问题是什么？
- 影响哪些用户？
- 发生在哪个流程节点？
- 有哪些证据？
- 哪些机会值得解决？
- 哪些方案成本最低、影响最大？

### 推荐应对流程

1. 描述现象：指标、反馈、流程或行为异常。
2. 定义问题：把“想改什么”转成“用户在哪个场景遇到什么障碍”。
3. 收集证据：数据、反馈、录屏、客服、访谈、日志。
4. 建立机会树：从业务目标到用户机会，再到方案和实验。
5. 发散方案：不要只保留第一个方案。
6. 识别假设：每个方案背后的风险是什么。
7. 优先级排序：按影响、成本、风险、战略匹配排序。
8. 输出改进 PRD 和测试场景。

### 必问问题

- 哪个用户群体受影响最大？
- 具体发生在哪个步骤？
- 当前基线指标是多少？
- 你希望改善哪个指标？
- 用户原话或数据证据是什么？
- 现在提出的方案解决的是症状，还是根因？

### 标准输出

- 问题定义。
- 当前基线和目标指标。
- 用户影响范围。
- 机会解决方案树。
- 方案对比和优先级。
- 改进 PRD。
- 验收标准和测试场景。

### 阻断条件

出现以下情况，不能进入方案或 PRD：

- 只有“想改版”，没有明确问题。
- 只有主观判断，没有用户或数据证据。
- 没有当前基线指标。
- 不知道目标用户是谁。
- 方案已经定了，但问题没有定义。

### 常见误区

- 把“页面不好看”当成问题定义。
- 把用户提的解决方案直接当需求。
- 没有基线就承诺提升效果。
- 只做单一方案，不做机会树和方案比较。
- 优化范围无限扩大。

### 示例提示词

```text
我们的新用户 onboarding 完成率只有 28%，用户反馈流程太长。请按已有产品问题优化场景处理：先帮我定义问题和机会树，再提出多个解决方案，排序后输出一个改进 PRD。
```

### 新人要掌握

已有产品优化先做诊断，再做方案。不要被第一个解决方案控制思路。

## S3：客户反馈和功能请求整理

### 场景定义

这是把大量杂乱反馈、客户需求、销售输入、客服工单、问卷和访谈片段整理成产品判断的场景。目标不是照单全收，而是识别主题、用户群、真实问题、优先级和路线图。

### 典型触发信号

- “客户提了很多需求。”
- “销售说这个客户必须要这个功能。”
- “帮我整理这些反馈。”
- “需求池太乱了。”
- “哪些客户需求应该排进路线图？”

### 适合使用

- 客户反馈很多，无法判断优先级。
- 需求来源分散在销售、客服、社群、问卷、访谈中。
- 需要把需求池整理成路线图。
- 需要识别客户群体差异。

### 不适合使用

- 只有单条高质量需求，直接进入 PRD 即可。
- 没有反馈原文或来源，只是二手转述。
- 已经完成优先级，只是要写开发任务。

### 关键判断

客户反馈需要从“声音”变成“证据”。关键判断包括：

- 反馈来自谁？
- 出现频次是多少？
- 代表哪个用户群？
- 背后的真实任务是什么？
- 是普遍问题还是大客户定制诉求？
- 和公司战略是否一致？
- 解决后带来什么业务影响？

### 推荐应对流程

1. 标准化输入：整理反馈来源、客户类型、原文、时间、频次。
2. 主题聚类：把相似反馈归并成问题主题。
3. 提炼真实诉求：区分用户说的功能和背后的任务。
4. 用户分群：识别不同群体的需求差异。
5. 评估价值：影响范围、客户价值、收入影响、留存影响。
6. 评估成本和风险。
7. 排优先级。
8. 输出 outcome roadmap，而不是简单功能列表。

### 必问问题

- 这些反馈来自哪些客户或用户群？
- 每类反馈出现频次是多少？
- 哪些反馈来自高价值客户？
- 这些需求对应哪个业务目标？
- 是否有替代方案或 workaround？
- 哪些请求是定制化，哪些是平台化能力？

### 标准输出

- 反馈原始清单。
- 主题聚类表。
- 用户群体分析。
- 需求背后的 JTBD。
- 优先级排序。
- 不做或暂缓的理由。
- 结果导向路线图。

### 阻断条件

出现以下情况，不能输出强优先级或路线图：

- 没有反馈来源。
- 没有频次或影响范围。
- 没有业务目标。
- 只有大客户声音，没有说明是否代表目标市场。
- 没有资源约束，却要求排期。

### 常见误区

- 谁声音大就先做谁。
- 把销售承诺直接变成产品路线图。
- 把定制需求误当平台需求。
- 只统计需求数量，不看客户价值和战略匹配。
- 没有说明为什么某些需求不做。

### 示例提示词

```text
下面是 80 条客户和销售反馈。请按客户反馈整理场景处理：先聚类主题，识别用户群体和真实诉求，再按影响、客户价值、成本、风险和战略匹配排序，最后输出 outcome roadmap。
```

### 新人要掌握

反馈是输入，不是结论。产品经理的价值在于把反馈转成判断和取舍。

## S4：写 PRD/需求文档

### 场景定义

这是把已经相对明确的产品问题和解决方案，整理成可评审、可设计、可开发、可测试、可验收文档的场景。PRD 是跨团队契约，不是功能描述清单。

### 典型触发信号

- “帮我写个需求。”
- “我要给研发提需求。”
- “帮我把这个功能写成 PRD。”
- “这个需求要进入排期。”
- “我们要开始做这个功能了。”

### 适合使用

- 用户、问题、目标和方案已基本明确。
- 需要设计或研发介入。
- 需要评审、排期、验收。
- 需要把口头需求转成正式文档。

### 不适合使用

- 目标用户不清楚。
- 问题没有定义。
- 只是一个想法，未做发现。
- 只有“改好看一点”“体验优化一下”这种模糊表达。

### 关键判断

进入 PRD 前必须确认：

- 用户是谁？
- 问题是什么？
- 为什么现在要做？
- 成功指标是什么？
- 范围内和范围外是什么？
- 用户流程是什么？
- 权限和边界条件是什么？
- 验收标准是什么？

### 推荐应对流程

1. PRD 前置检查：判断是否具备写 PRD 的条件。
2. 明确背景、目标和问题。
3. 定义用户细分和使用场景。
4. 说明解决方案和范围。
5. 写用户故事或 job stories。
6. 写验收标准。
7. 生成测试场景。
8. 做 pre-mortem 风险分析。
9. 输出正式 PRD。

### 必问问题

- 这个需求服务哪个用户和场景？
- 要解决的用户问题是什么？
- 业务目标和成功指标是什么？
- 这个版本必须包含什么？
- 明确不做什么？
- 哪些状态、权限、异常和边界条件必须覆盖？
- 研发完成后如何验收？

### 标准输出

- PRD。
- 用户故事。
- Job stories 或 WWA。
- 验收标准。
- 测试场景。
- 风险和依赖。
- Open questions。

### 阻断条件

出现以下情况，不能输出正式 PRD，只能输出问题清单：

- 用户不清楚。
- 问题不清楚。
- 成功指标不清楚。
- 范围边界不清楚。
- 验收标准不清楚。
- 需求里只有方案，没有背景和目标。

### 常见误区

- PRD 只有功能点，没有用户问题。
- 没有范围外事项。
- 没有验收标准。
- 没有异常状态和边界条件。
- 把 UI 细节当作需求本身。

### 示例提示词

```text
请按 PRD 标准化场景处理下面这个需求。先检查信息是否足够写 PRD，如果不足请阻断并追问；如果足够，请输出 PRD、用户故事、验收标准、测试场景和风险清单。
```

### 新人要掌握

PRD 是交付契约。它必须让设计知道要设计什么，让研发知道要实现什么，让测试知道怎么验收，让业务知道为什么做。

## S5：用户研究和访谈

### 场景定义

这是通过访谈、问卷、观察、用户记录等方式理解用户行为、痛点、任务和决策过程的场景。它可以发生在新产品探索、已有产品优化、定价、定位、上线复盘等多个阶段。

### 典型触发信号

- “我要访谈用户。”
- “帮我写访谈提纲。”
- “这段访谈帮我总结。”
- “我们想了解用户为什么不用。”
- “帮我从访谈中提炼需求。”

### 适合使用

- 需要理解用户真实行为。
- 需要验证问题是否存在。
- 需要理解购买、使用、流失原因。
- 需要从访谈记录中提炼洞察。

### 不适合使用

- 已经有明确数据结论，只需要算指标。
- 只是要问用户是否喜欢某个方案。
- 研究问题不明确。

### 关键判断

用户研究的质量取决于研究问题。必须先明确：

- 这次研究要支持什么决策？
- 访谈对象是谁？
- 要理解过去行为还是未来偏好？
- 需要避免哪些引导性问题？
- 输出要支持产品发现、PRD、定位、定价还是上线复盘？

### 推荐应对流程

1. 明确研究目标和决策用途。
2. 定义访谈对象和筛选条件。
3. 设计非引导性问题。
4. 访谈后提炼 JTBD、痛点、替代方案、触发因素。
5. 区分事实、原话、解释和假设。
6. 形成洞察和后续行动。
7. 必要时生成用户画像或分群。

### 必问问题

- 这次访谈要帮助你做什么决策？
- 要访谈哪类用户？为什么是他们？
- 你最不确定的问题是什么？
- 你希望了解过去行为、当前痛点还是购买决策？
- 访谈结束后要产出什么？

### 标准输出

- 研究目标。
- 访谈对象标准。
- 访谈提纲。
- 访谈摘要。
- 用户原话和证据。
- JTBD、痛点、替代方案。
- 洞察和行动项。

### 阻断条件

出现以下情况，不能生成正式访谈提纲：

- 不知道研究要支持什么决策。
- 不知道访谈对象是谁。
- 问题是引导性的。
- 只是问“你喜不喜欢这个功能”。

访谈总结时，如果没有原话或证据，只能标记为假设。

### 常见误区

- 问用户未来会不会用。
- 直接展示方案并寻求认可。
- 把单个用户观点当普遍结论。
- 总结只有观点，没有证据。
- 访谈后没有行动项。

### 示例提示词

```text
我们想了解企业管理员为什么没有完成权限配置。请按用户访谈准备场景帮我设计 45 分钟访谈提纲，遵循不引导、不推销、聚焦过去行为的原则。
```

```text
下面是 5 段访谈记录。请按访谈总结场景提炼 JTBD、痛点、替代方案、满意度信号、关键原话和产品行动项。
```

### 新人要掌握

访谈不是让用户替你设计产品，而是理解用户真实行为和决策上下文。

## S6：功能优先级和路线图

### 场景定义

这是在资源有限的情况下，对多个功能、机会、需求或项目进行排序，并形成路线图的场景。核心是取舍，而不是把所有东西排进计划。

### 典型触发信号

- “这么多功能先做哪个？”
- “下季度路线图怎么排？”
- “资源不够，怎么取舍？”
- “老板、销售、客户都要不同东西。”
- “帮我排一下优先级。”

### 适合使用

- 候选功能很多。
- 团队资源有限。
- 需要季度或月度规划。
- 需要向管理层解释取舍。
- 需要从功能路线图转成结果路线图。

### 不适合使用

- 没有明确业务目标。
- 候选项还没有基本定义。
- 只是单个需求写 PRD。

### 关键判断

优先级必须基于统一目标和口径。要先明确：

- 当前最重要的业务目标是什么？
- 每个功能影响哪个用户和指标？
- 影响范围和证据是什么？
- 成本和依赖是什么？
- 风险是什么？
- 什么不做？为什么？

### 推荐应对流程

1. 明确业务目标和规划周期。
2. 统一候选项描述。
3. 选择优先级框架，如 RICE、ICE、MoSCoW、Kano 等。
4. 定义评分口径。
5. 对候选项评分。
6. 输出排序和取舍理由。
7. 改写成 outcome roadmap。
8. 记录关键决策。

### 必问问题

- 这次排序服务哪个业务目标？
- 规划周期是多长？
- 资源约束是什么？
- 每个候选项的用户价值和业务价值是什么？
- 哪些是必须做，哪些是可延后？
- 谁需要参与或认可这个取舍？

### 标准输出

- 候选项列表。
- 优先级框架选择理由。
- 评分表。
- Top priorities。
- 暂缓和不做清单。
- Outcome roadmap。
- 决策记录。

### 阻断条件

出现以下情况，不能输出正式路线图：

- 没有业务目标。
- 没有统一评分口径。
- 候选项定义不清。
- 没有资源约束。
- 没有说明取舍理由。

### 常见误区

- 谁级别高就先做谁的需求。
- 只看收入，不看成本和战略。
- 只排功能，不写 outcome。
- 路线图承诺过细时间，但资源和依赖没确认。
- 没有记录为什么不做某些功能。

### 示例提示词

```text
下面有 18 个候选功能。请按功能优先级和路线图场景处理：先确认业务目标和评分口径，再用合适框架排序，最后输出 outcome roadmap 和不做/暂缓理由。
```

### 新人要掌握

优先级的本质是取舍。好的路线图讲清楚要达成什么结果，而不是只列要做什么功能。

## S7：指标和增长实验

### 场景定义

这是定义产品指标、搭建指标看板、设计增长实验、分析 A/B 测试和 cohort 留存的场景。目标是让产品工作从“做了什么”转成“产生了什么结果”。

### 典型触发信号

- “我们该看什么指标？”
- “北极星指标怎么定？”
- “这个实验能不能上线？”
- “留存下降了怎么分析？”
- “帮我写 SQL 看这个指标。”
- “我们要做增长实验。”

### 适合使用

- 新产品或新功能需要定义成功指标。
- 已有产品需要建立指标体系。
- 增长团队要设计实验。
- 需要分析 A/B 测试结果。
- 需要 cohort 留存和行为分析。

### 不适合使用

- 没有产品目标，只想随便看数据。
- 数据口径完全不清。
- 样本量不足却要求做确定结论。

### 关键判断

指标工作先判断：

- 产品交付的用户价值是什么？
- 业务模式是什么：Attention、Transaction、Productivity？
- 北极星指标是否反映用户价值？
- 输入指标是否可行动？
- 护栏指标是否防止副作用？
- 实验是否有假设、样本量、周期和决策标准？

### 推荐应对流程

1. 明确产品目标和业务模式。
2. 定义北极星指标。
3. 拆输入指标、健康指标、护栏指标。
4. 设计指标看板。
5. 明确数据源和口径。
6. 生成 SQL 或分析方案。
7. 设计实验。
8. 分析实验结果，给出 ship、extend 或 stop 建议。

### 必问问题

- 产品为用户创造的核心价值是什么？
- 当前业务目标是什么？
- 用户完成什么行为代表获得价值？
- 数据从哪里来？口径是什么？
- 当前基线是多少？
- 实验假设是什么？
- 决策标准是什么？

### 标准输出

- 北极星指标。
- 输入指标。
- 健康指标和护栏指标。
- 指标口径。
- 看板设计。
- SQL 查询。
- 实验计划。
- 实验分析结论。

### 阻断条件

出现以下情况，不能给出正式指标或实验结论：

- 指标不能代表用户价值。
- 没有清晰口径。
- 没有基线。
- 实验没有样本量或周期说明。
- A/B 测试没有显著性和置信区间分析。

### 常见误区

- 把 DAU、PV 直接当北极星。
- 只看增长指标，不设护栏指标。
- 实验没跑够就宣布胜利。
- 数据口径不一致。
- 指标太多，没有行动优先级。

### 示例提示词

```text
我们是一个 B2B 项目协作 SaaS，想重新定义产品指标。请按指标体系场景处理：先判断业务游戏类型，再设计北极星指标、输入指标、健康指标、护栏指标和看板。
```

```text
Control 转化率 4.2%，n=5000；Variant 转化率 4.8%，n=5100。请按 A/B 实验分析场景判断是否应该上线、继续实验还是停止。
```

### 新人要掌握

指标不是报表装饰，而是产品决策工具。好指标必须能反映用户价值，并能指导行动。

## S8：产品战略和商业模式

### 场景定义

这是判断产品方向、目标市场、价值主张、商业模式、竞争环境、定价和增长路径的场景。它通常服务于年度规划、融资、市场进入、产品线调整或重大方向选择。

### 典型触发信号

- “这个方向值不值得做？”
- “我们的产品战略是什么？”
- “商业模式怎么定？”
- “这个市场有多大？”
- “我们怎么和竞品区分？”
- “定价应该怎么做？”

### 适合使用

- 产品方向不清。
- 新市场进入。
- 年度或季度战略规划。
- 商业模式和定价探索。
- 需要竞品、市场和宏观环境分析。

### 不适合使用

- 单个小功能需求。
- 只是写开发任务。
- 没有明确市场或用户范围。

### 关键判断

战略要回答：

- 我们服务谁？
- 他们为什么需要我们？
- 我们和替代方案相比强在哪里？
- 这个市场是否值得做？
- 我们怎么赚钱？
- 我们选择做什么，不做什么？
- 我们的增长路径和护城河是什么？

### 推荐应对流程

1. 明确战略问题。
2. 定义产品愿景。
3. 明确目标用户和价值主张。
4. 做市场规模估算。
5. 做竞品和差异化分析。
6. 做 SWOT、PESTLE、五力等战略扫描。
7. 设计商业模式和定价方向。
8. 输出产品战略画布。
9. 转成 OKR 和 outcome roadmap。

### 必问问题

- 这次战略分析要支持什么决策？
- 目标市场和目标用户是谁？
- 用户当前替代方案是什么？
- 我们的差异化能力是什么？
- 商业模式约束是什么？
- 成功的时间尺度是多久？
- 哪些方向明确不做？

### 标准输出

- 产品愿景。
- 产品战略画布。
- 价值主张。
- 市场规模。
- 竞品和差异化。
- 商业模式。
- 定价方向。
- 战略取舍。
- OKR 和路线图。

### 阻断条件

出现以下情况，不能输出正式战略：

- 不知道目标市场。
- 不知道目标用户。
- 不知道替代方案。
- 没有战略问题，只是泛泛分析。
- 战略无法说明“不做什么”。
- 商业模式和增长路径完全缺失。

### 常见误区

- 把愿景口号当战略。
- 战略只有目标，没有取舍。
- 只做竞品功能对比，不分析价值主张。
- 市场规模拍脑袋。
- 定价只参考竞品，不看支付意愿和价值。

### 示例提示词

```text
我们考虑进入中小企业 AI 客服市场。请按产品战略和商业模式场景处理：输出目标用户、市场规模、竞品分析、价值主张、商业模式、定价方向、战略取舍和初步路线图。
```

### 新人要掌握

战略不是“我们想成为谁”，而是“我们选择服务谁、创造什么价值、通过什么方式取胜、明确不做什么”。

## S9：上线发布和风险检查

### 场景定义

这是产品或功能准备上线前的检查场景。目标是确认风险、测试、沟通、监控、回滚、发布说明和相关方准备是否到位。

### 典型触发信号

- “准备上线了，帮我检查。”
- “这个功能下周发布。”
- “帮我写发布说明。”
- “上线有什么风险？”
- “这个需求能不能发？”

### 适合使用

- 功能即将上线。
- 重大客户交付。
- 涉及权限、数据、支付、通知、自动化等风险。
- 需要发布说明和沟通计划。
- 需要上线前风险预案。

### 不适合使用

- 需求还没定义清楚。
- 只是早期想法探索。
- 没有研发完成或测试结果。

### 关键判断

上线前要判断：

- 核心用户路径是否可用？
- 验收标准是否全部通过？
- 是否有 launch-blocking 风险？
- 是否有监控指标？
- 是否有回滚方案？
- 相关方是否同步？
- 用户和客服是否知道变化？

### 推荐应对流程

1. 对照 PRD 和验收标准检查完成情况。
2. 做 pre-mortem，想象上线失败的原因。
3. 做 red-team，攻击关键假设。
4. 检查测试场景覆盖。
5. 制定 stakeholder 沟通计划。
6. 写发布说明。
7. 定义监控和回滚。
8. 输出上线 checklist。

### 必问问题

- 哪些用户会受影响？
- 哪些核心路径必须正常？
- 测试通过情况如何？
- 如果失败，最严重后果是什么？
- 如何监控上线效果？
- 谁有权决定发布或回滚？
- 客服、销售、运营是否需要同步？

### 标准输出

- 上线风险清单。
- Launch-blocking / fast-follow / track 分类。
- 测试覆盖清单。
- 发布说明。
- 相关方沟通计划。
- 监控指标。
- 回滚方案。
- 上线 checklist。

### 阻断条件

出现以下情况，不能上线：

- 有 launch-blocking 风险未解决。
- 核心路径未测试。
- 没有回滚方案。
- 没有监控指标。
- 权限或数据风险未确认。
- 关键相关方未同步。

### 常见误区

- 研发完成等于可以上线。
- 只写发布说明，不做风险检查。
- 没有回滚预案。
- 没有客服和销售准备。
- 上线后不看指标。

### 示例提示词

```text
这个自助开票功能准备上线。请按上线发布和风险检查场景处理：先做 pre-mortem 和 red-team，再检查测试场景、监控指标、回滚方案、相关方沟通和发布说明。
```

### 新人要掌握

上线是产品责任的一部分。真正的完成不是“功能做完”，而是“用户可用、风险可控、团队知道、指标可追踪”。

## S10：PM 日常运转

### 场景定义

这是产品经理日常协作和运营场景，包括会议纪要、OKR、Sprint 计划、复盘、假数据、行动项和决策记录。它看似琐碎，但决定团队执行质量。

### 典型触发信号

- “帮我整理会议纪要。”
- “帮我写 OKR。”
- “下个 Sprint 怎么排？”
- “帮我做复盘。”
- “生成一些测试数据。”
- “把行动项整理一下。”

### 适合使用

- 产品会议、评审会、复盘会。
- 团队 OKR 制定。
- Sprint 计划和复盘。
- 需要沉淀决策和行动项。
- 需要生成测试或演示数据。

### 不适合使用

- 重大新产品方向，应该进入战略或发现工作流。
- 准备进入研发的正式需求，应该进入 PRD 标准化。
- 上线前风险检查，应该进入 S9。

### 关键判断

日常工作不能只产出“记录”，还要产出：

- 决策是什么？
- 行动项是什么？
- 谁负责？
- 截止时间是什么？
- 有哪些未决问题？
- 是否影响路线图或需求？

### 推荐应对流程

1. 判断任务类型：会议、OKR、Sprint、复盘、数据。
2. 提取背景和目标。
3. 结构化输出。
4. 标出决策、行动项、owner、deadline。
5. 记录 open questions。
6. 如果产生产品决策，写入 decision log。

### 必问问题

- 这份材料服务谁？
- 需要输出决策、行动项还是计划？
- 有哪些 owner 和截止时间？
- 哪些事项需要后续跟进？
- 是否影响已有 PRD、路线图或上线计划？

### 标准输出

- 会议纪要。
- 决策列表。
- 行动项和 owner。
- OKR。
- Sprint 计划。
- 复盘行动项。
- 测试数据。
- Decision log。

### 阻断条件

出现以下情况，不能算完成：

- 会议纪要没有决策。
- 行动项没有 owner。
- 行动项没有截止时间。
- OKR 没有可衡量 KR。
- Sprint 计划没有容量、依赖和风险。
- 复盘没有后续行动。

### 常见误区

- 会议纪要只是流水账。
- OKR 写成任务列表。
- Sprint 计划没有容量估算。
- 复盘只说问题，不定行动项。
- 决策散落在聊天记录里，没有沉淀。

### 示例提示词

```text
请按 PM 日常运转场景整理下面这段会议记录。输出会议主题、关键讨论、已做决策、行动项、owner、截止时间、open questions 和需要写入 decision log 的事项。
```

```text
请帮增长团队制定下季度 OKR。公司目标是提升付费转化，团队负责 onboarding 和激活。请输出 3 组 Objective 和可衡量 KR。
```

### 新人要掌握

PM 日常工作的价值不是“把事情记下来”，而是让决策、责任、时间和后续行动变清楚。

## 场景判断练习

下面这些输入应该如何判断：

| 用户输入 | 推荐场景 | 原因 |
| --- | --- | --- |
| “我们想做一个 AI 招聘助手。” | S1 | 新产品想法，目标用户和问题待澄清 |
| “首页转化率下降了，帮我优化。” | S2 | 已有产品指标问题，需要诊断 |
| “这里有 200 条销售反馈，帮我排需求。” | S3 | 大量反馈和需求整理 |
| “帮我给研发写个需求。” | S4 | PRD/需求文档，但要先检查信息完整度 |
| “我要访谈 10 个管理员用户。” | S5 | 用户研究和访谈 |
| “下季度这 15 个功能先做哪个？” | S6 | 功能优先级和路线图 |
| “我们的北极星指标该是什么？” | S7 | 指标体系 |
| “这个市场值得进入吗？” | S8 | 战略和商业模式 |
| “功能做完了，准备上线。” | S9 | 上线发布和风险检查 |
| “帮我整理这次评审会纪要。” | S10 | PM 日常运转 |

## 通用应对原则

1. 任何场景都先判断，不直接输出最终文档。
2. 任何需求都要区分事实、假设、观点。
3. 任何 PRD 都必须有用户、问题、目标、范围、验收标准。
4. 任何优先级都必须有目标和评分口径。
5. 任何上线都必须有风险、测试、监控和回滚。
6. 任何会议纪要都必须有决策和行动项。
7. 信息不足时，必须阻断，而不是替用户脑补。

## 建议的团队学习方式

建议团队用 3 次培训掌握这些场景：

### 第 1 次：认识场景

目标：让大家知道产品工作不等于写需求。

内容：

- 讲解十个场景。
- 每个场景给一个真实例子。
- 做场景判断练习。

### 第 2 次：掌握阻断

目标：让大家学会什么时候不能进入下一步。

内容：

- 讲解通用阻断条件。
- 对低质量需求做 Gate Review。
- 练习如何追问。

### 第 3 次：产出标准文档

目标：让大家能按规范输出。

内容：

- 用一个真实需求跑完整流程。
- 输出 Product Brief、PRD、路线图或发布检查。
- 做同伴评审。

## 最小要求

所有人至少要做到：

- 能判断十个场景。
- 能知道每个场景的第一步不是写最终文档。
- 能识别什么情况下必须阻断。
- 能按模板输出标准产物。
- 能把不确定性写成假设和 open questions。

如果做到这些，即使是刚接触产品工作的同事，也能用统一方法推进产品工作，而不是凭感觉写文档。
